### **1. Middleware in Express.js**

> **Level:** Intermediate

#### Why This Is Asked
Tests whether the candidate understands Express's request-response lifecycle and can reason about ordering, composability, and error propagation in middleware chains.

#### Explanation
Middleware are functions with access to `req`, `res`, and `next`. They execute sequentially in the order they are registered. Each middleware either ends the request by sending a response, or calls `next()` to hand off to the next function in the chain. Express also supports a special four-argument signature `(err, req, res, next)` for error-handling middleware, which only activates when `next(err)` is called.

#### Code Example
```javascript
const express = require('express');
const app = express();

// Logging middleware
app.use((req, res, next) => {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next();
});

// Route handler
app.get('/', (req, res) => {
  res.send('Hello, Middleware!');
});

// Error-handling middleware ‚Äî must be last, must have 4 arguments
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).send('Something went wrong');
});

app.listen(3000);
```

#### Do's and Don'ts

**Do's**
- Always call `next()` or send a response ‚Äî leaving the request hanging causes client timeouts.
- Register error-handling middleware last with four arguments `(err, req, res, next)`.
- Keep each middleware focused on a single concern (logging, auth, parsing).

**Don'ts**
- Don't assume middleware order doesn't matter ‚Äî Express executes them top to bottom.
- Avoid CPU-intensive work inside middleware; offload to a worker thread or queue.
- Don't call `next()` after you've already sent a response ‚Äî it triggers `Cannot set headers after they are sent`.

#### Common Follow-up Questions
- What happens if you call `next(err)` inside a middleware?
- How would you apply middleware to only a specific route, not all routes?
- What is the difference between `app.use()` and `router.use()`?

### **2. Difference Between npm and npx**

> **Level:** Beginner

#### Why This Is Asked
Checks foundational tooling knowledge ‚Äî understanding the Node.js ecosystem is a baseline expectation for any Node.js developer.

#### Explanation
`npm` (Node Package Manager) installs, updates, and manages packages for a project or globally on your machine. `npx` is a package runner bundled with npm since v5.2 ‚Äî it executes a package binary without requiring a global install. If the binary is not already installed, `npx` downloads it temporarily, runs it, then discards it, keeping your global environment clean.

#### Code Example
```bash
# npm: installs globally and persists on disk
npm install -g typescript
tsc --version

# npx: runs without a global install (downloads, runs, discards)
npx create-react-app my-app

# npx also runs a project-local binary (avoids version conflicts)
npx jest --watchAll
npx eslint src/
```

#### Do's and Don'ts

**Do's**
- Use `npx` for one-off CLI tools to avoid polluting the global namespace.
- Use `npx` to always run the project-local version of a tool, ensuring consistent versions across the team.
- Pin a specific version with `npx pkg@version` when reproducibility matters.

**Don'ts**
- Don't use `npx` for packages your app depends on at runtime ‚Äî those belong in `dependencies`.
- Avoid global installs with `-g` for project tooling; prefer `devDependencies` + `npx`.

#### Common Follow-up Questions
- What is the difference between `dependencies`, `devDependencies`, and `peerDependencies`?
- How does `npm ci` differ from `npm install`, and when should each be used?
- Why might a team prefer `pnpm` or `yarn` over `npm`?

### **3. Using WebSockets in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Tests understanding of real-time communication patterns and when to use persistent connections over request-response HTTP.

#### Explanation
WebSockets provide a persistent, full-duplex TCP connection between client and server. Unlike HTTP, once the connection is established via an upgrade handshake, both sides can push messages at any time without polling. The `ws` library is the lightweight, production-ready choice in Node.js. WebSockets are ideal for chat, live dashboards, and multiplayer games ‚Äî use HTTP for everything else.

#### Code Example
```javascript
const WebSocket = require('ws');
const server = new WebSocket.Server({ port: 8080 });

server.on('connection', (socket) => {
  console.log('Client connected');

  socket.on('message', (msg) => {
    console.log(`Received: ${msg}`);
    socket.send(`Server echo: ${msg}`);
  });

  socket.on('close', () => console.log('Client disconnected'));

  // Always handle errors to prevent uncaught exceptions
  socket.on('error', (err) => console.error('Socket error:', err));
});

// Heartbeat: detect and clean up stale connections
const interval = setInterval(() => {
  server.clients.forEach((ws) => {
    if (!ws.isAlive) return ws.terminate();
    ws.isAlive = false;
    ws.ping();
  });
}, 30000);

server.on('close', () => clearInterval(interval));
```

#### Do's and Don'ts

**Do's**
- Always handle the `error` and `close` events on every socket to prevent uncaught exceptions.
- Use `wss://` (WebSocket Secure) in production ‚Äî terminate TLS at a load balancer or proxy.
- Implement heartbeats (ping/pong) to detect and clean up zombie connections.

**Don'ts**
- Don't send large binary payloads without chunking ‚Äî it can stall the socket.
- Avoid WebSockets for simple request-response patterns; HTTP is simpler and cache-friendly.
- Don't neglect reconnection logic on the client ‚Äî networks drop unexpectedly.

#### Common Follow-up Questions
- How does the WebSocket upgrade handshake work at the HTTP protocol level?
- When would you choose WebSockets over Server-Sent Events (SSE)?
- How would you scale a WebSocket server across multiple Node.js instances?

### **4. Purpose of Query String Module**

> **Level:** Beginner

#### Why This Is Asked
Assesses familiarity with URL handling in Node.js and whether the candidate is aware of modern vs. legacy APIs.

#### Explanation
The `querystring` module parses and serialises URL query parameters. It is now considered legacy ‚Äî Node.js officially recommends the WHATWG `URLSearchParams` API (available globally since Node.js v10), which is browser-compatible and integrates cleanly with the `URL` class. New code should use `URL` and `URLSearchParams` exclusively.

#### Code Example
```javascript
// Legacy: querystring module (still works, but deprecated direction)
const querystring = require('querystring');
const parsed = querystring.parse('name=John&age=25');
console.log(parsed); // { name: 'John', age: '25' }

// Modern: URLSearchParams (preferred)
const params = new URLSearchParams('name=John&age=25');
console.log(params.get('name')); // 'John'
console.log(Object.fromEntries(params)); // { name: 'John', age: '25' }

// Parsing a full URL with query string
const url = new URL('https://example.com/api?name=John&age=25');
console.log(url.searchParams.get('age')); // '25'
url.searchParams.set('page', '2');
console.log(url.toString()); // https://example.com/api?name=John&age=25&page=2
```

#### Do's and Don'ts

**Do's**
- Prefer `URLSearchParams` and the `URL` class for all new code ‚Äî they are standards-based and work in both Node.js and browsers.
- Use `URLSearchParams` to construct query strings safely ‚Äî it handles encoding automatically.

**Don'ts**
- Don't manually concatenate query strings with string interpolation ‚Äî it creates encoding bugs for special characters.
- Avoid mixing `querystring` and `URLSearchParams` in the same codebase.

#### Common Follow-up Questions
- How does `URLSearchParams` handle duplicate keys (e.g., `?tag=a&tag=b`)?
- How would you safely construct a URL with dynamic user-provided parameters?

### **5. Using the Events Module**

> **Level:** Intermediate

#### Why This Is Asked
Node.js is fundamentally event-driven ‚Äî this question checks whether the candidate understands the core pattern underlying streams, HTTP servers, and most of the Node ecosystem.

#### Explanation
The `EventEmitter` class is the backbone of Node.js's event-driven architecture. Streams, `http.Server`, `child_process`, and most core APIs extend `EventEmitter`. Listeners registered with `.on()` are called synchronously in registration order when the event fires. `EventEmitter` is also a powerful pattern for building decoupled, observable services in your own code.

#### Code Example
```javascript
const EventEmitter = require('events');

class UserService extends EventEmitter {
  createUser(name) {
    // ... create user in DB
    this.emit('userCreated', { name, createdAt: new Date() });
  }
}

const service = new UserService();

// Persistent listener ‚Äî fires on every event
service.on('userCreated', (user) => {
  console.log(`New user registered: ${user.name}`);
});

// One-time listener ‚Äî fires once then removes itself
service.once('userCreated', () => {
  console.log('First user ever created ‚Äî send welcome email to admin');
});

service.createUser('Alice'); // Both listeners fire
service.createUser('Bob');   // Only the persistent listener fires
```

#### Do's and Don'ts

**Do's**
- Remove listeners with `.off()` or `.removeListener()` when the emitter's lifecycle ends to prevent memory leaks.
- Use `.once()` for events that should trigger a single one-time action.
- Always attach an `'error'` event listener ‚Äî emitting `'error'` without one throws an uncaught exception.

**Don'ts**
- Don't register unbounded numbers of listeners ‚Äî Node.js warns at 10 (configurable via `setMaxListeners`).
- Avoid heavy synchronous work inside event handlers ‚Äî it blocks the event loop for all other events.
- Don't emit `'error'` without a listener attached.

#### Common Follow-up Questions
- What happens when you emit an `'error'` event with no listener?
- What is the difference between `.on()` and `.once()`?
- How does back-pressure work in a `Readable` stream, which extends `EventEmitter`?

### **6. Async/Await in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Async/await is the dominant pattern for asynchronous code in modern Node.js ‚Äî interviewers use this to confirm you can reason about error propagation, parallelism, and common pitfalls like sequential awaits.

#### Explanation
`async` functions always return a Promise. Inside them, `await` pauses execution of that function until the awaited Promise settles, unwrapping the resolved value or re-throwing on rejection. Async/await is syntactic sugar over Promises ‚Äî it does not create threads. The most common performance mistake is using `await` sequentially for independent operations that could run in parallel with `Promise.all`.

#### Code Example
```javascript
// Anti-pattern: sequential awaits for independent operations
const bad = async (userId) => {
  const user = await db.users.findById(userId);       // waits...
  const orders = await db.orders.findByUser(userId);  // then waits...
  return { user, orders };
};

// Correct: run independent calls in parallel
const good = async (userId) => {
  const [user, orders, notifications] = await Promise.all([
    db.users.findById(userId),
    db.orders.findByUser(userId),
    db.notifications.findByUser(userId),
  ]);
  return { user, orders, notifications };
};

// Always handle errors at the boundary
const main = async () => {
  try {
    const dashboard = await good(42);
    console.log(dashboard);
  } catch (err) {
    console.error('Dashboard load failed:', err.message);
  }
};

main();
```

#### Do's and Don'ts

**Do's**
- Use `Promise.all()` to run independent async operations in parallel.
- Use `Promise.allSettled()` when you need all results even if some fail.
- Always wrap top-level `async` calls in `try/catch` or attach `.catch()` to handle unhandled rejections.

**Don'ts**
- Don't `await` inside `forEach` ‚Äî it doesn't wait for each iteration. Use `for...of` or `Promise.all` with `.map()`.
- Avoid unnecessary sequential `await` chains for operations that don't depend on each other.
- Don't silently swallow errors in `catch` blocks ‚Äî at minimum, log them.

#### Common Follow-up Questions
- What is the difference between `Promise.all` and `Promise.allSettled`?
- What happens if you `await` a non-Promise value?
- How do you cancel an in-flight async operation in Node.js?

### **7. The Event Loop**

> **Level:** Intermediate

#### Why This Is Asked
The event loop is the core of Node.js's concurrency model ‚Äî understanding it separates candidates who know Node.js deeply from those who only use it superficially.

#### Explanation
Node.js runs on a single thread but achieves concurrency through the event loop, which cycles through phases: **timers** (`setTimeout`, `setInterval`) ‚Üí **pending callbacks** ‚Üí **poll** (I/O events) ‚Üí **check** (`setImmediate`) ‚Üí **close callbacks**. Between each phase, Node.js drains two queues with higher priority: the **nextTick queue** (`process.nextTick`) and the **microtask queue** (resolved Promises, `queueMicrotask`). The nextTick queue is drained before microtasks.

#### Code Example
```javascript
console.log('1 - synchronous');

setTimeout(() => console.log('2 - setTimeout (timers phase)'), 0);

Promise.resolve().then(() => console.log('3 - Promise microtask'));

process.nextTick(() => console.log('4 - process.nextTick'));

setImmediate(() => console.log('5 - setImmediate (check phase)'));

console.log('6 - synchronous');

// Output order:
// 1 - synchronous
// 6 - synchronous
// 4 - process.nextTick      ‚Üê nextTick queue (highest priority)
// 3 - Promise microtask     ‚Üê microtask queue
// 2 - setTimeout            ‚Üê timers phase
// 5 - setImmediate          ‚Üê check phase
```

#### Do's and Don'ts

**Do's**
- Use `setImmediate` to defer non-urgent work to after the current I/O events have been handled.
- Use profiling tools like `clinic.js` or `0x` to measure and diagnose event loop lag.
- Break up long synchronous operations with `setImmediate` or `process.nextTick` to yield back to the loop.

**Don'ts**
- Don't run CPU-intensive synchronous code on the main thread ‚Äî it blocks all I/O for the duration.
- Avoid deeply recursive `process.nextTick` calls ‚Äî they starve I/O by never letting the loop advance.
- Don't treat `setTimeout(fn, 0)` as "immediate" ‚Äî it runs after I/O events, which can delay it.

#### Common Follow-up Questions
- What is the difference between `process.nextTick` and `setImmediate`?
- How does Node.js handle I/O without multiple threads?
- What tools would you use to diagnose event loop lag in production?

### **8. Optimizing Performance in Node.js**

> **Level:** Advanced

#### Why This Is Asked
Tests whether the candidate thinks in systems ‚Äî they should know where Node.js bottlenecks actually occur and what the correct lever is for each type of problem.

#### Explanation
Node.js performance problems fall into two categories: **I/O-bound** (the event loop is waiting on disk, network, or DB ‚Äî fix with caching, connection pooling, or streaming) and **CPU-bound** (the event loop is blocked by computation ‚Äî fix with `worker_threads`, child processes, or offloading to a dedicated service). Always profile before optimising ‚Äî guessing is expensive.

#### Code Example
```javascript
const cluster = require('cluster');
const http = require('http');
const os = require('os');

// Scale across all CPU cores
if (cluster.isPrimary) {
  for (let i = 0; i < os.cpus().length; i++) cluster.fork();

  cluster.on('exit', (worker) => {
    console.warn(`Worker ${worker.process.pid} died ‚Äî restarting`);
    cluster.fork();
  });
} else {
  // In-process cache ‚Äî use Redis in multi-instance deployments
  const cache = new Map();

  http.createServer(async (req, res) => {
    if (cache.has(req.url)) {
      res.end(cache.get(req.url));
      return;
    }
    const data = await fetchFromDatabase(req.url);
    cache.set(req.url, data);
    res.end(data);
  }).listen(3000);
}
```

#### Do's and Don'ts

**Do's**
- Profile first ‚Äî use `clinic.js`, `0x`, or Node's built-in `--prof` flag to find the real bottleneck.
- Use `worker_threads` for CPU-bound work so the main thread stays free for I/O.
- Enable HTTP keep-alive and connection pooling for database and downstream service connections.
- Stream large responses instead of buffering the entire payload in memory.

**Don'ts**
- Don't use an in-process `Map` as a cache in multi-instance deployments ‚Äî use Redis.
- Avoid heavy synchronous `JSON.parse` / `JSON.stringify` on large objects in hot paths.
- Don't log verbosely in production ‚Äî high-volume logging I/O becomes a measurable bottleneck.

#### Common Follow-up Questions
- What is the difference between `cluster` and `worker_threads`, and when would you use each?
- How would you detect and diagnose an event loop lag spike in production?
- What does connection pooling solve, and when is it essential?

### **9. Handling CORS in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
CORS is a frequent source of bugs in full-stack apps ‚Äî interviewers want to verify you understand *why* it exists and what it actually controls, not just that you know which npm package to install.

#### Explanation
CORS (Cross-Origin Resource Sharing) is a browser security mechanism. Browsers block cross-origin reads by default to prevent malicious sites from making authenticated requests to another origin on behalf of the user. For simple requests, the browser sends an `Origin` header and the server must respond with a matching `Access-Control-Allow-Origin`. For non-simple requests (custom headers, `PUT`/`DELETE`), the browser sends an HTTP `OPTIONS` **preflight** request first. CORS is enforced by browsers only ‚Äî it does not protect against server-to-server calls or `curl`.

#### Code Example
```javascript
const express = require('express');
const cors = require('cors');
const app = express();

// Production-grade CORS configuration
const corsOptions = {
  origin: ['https://myapp.com', 'https://staging.myapp.com'],
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true,  // allow cookies / Authorization headers
  maxAge: 86400,      // cache preflight for 24 hours
};

app.use(cors(corsOptions));

// Explicitly handle preflight for all routes
app.options('*', cors(corsOptions));

app.get('/api/data', (req, res) => {
  res.json({ message: 'CORS-configured response' });
});
```

#### Do's and Don'ts

**Do's**
- Restrict `origin` to an explicit allowlist in production rather than allowing all origins.
- Cache preflight responses with `maxAge` to reduce round-trips from repeated OPTIONS requests.
- Set `credentials: true` only when your app actually needs cookies or `Authorization` headers.

**Don'ts**
- Never use `origin: '*'` with `credentials: true` ‚Äî browsers block this combination by spec.
- Don't disable CORS as a debugging shortcut and forget to re-enable it before shipping.
- Avoid treating CORS as a security control for server-to-server traffic ‚Äî it only applies to browsers.

#### Common Follow-up Questions
- What is a preflight request and exactly when does the browser trigger one?
- Why does CORS not prevent server-to-server requests or `curl` requests?
- How does setting `credentials: true` change what `origin` values are permitted?

### **10. Real-Time Applications with Socket.io**

> **Level:** Intermediate

#### Why This Is Asked
Socket.io is widely used in Node.js real-time applications ‚Äî this tests practical knowledge of event-driven client-server communication and the trade-offs of Socket.io versus raw WebSockets.

#### Explanation
Socket.io wraps WebSockets with automatic fallback to long-polling, built-in rooms and namespaces, reconnection logic, and event multiplexing. It uses a custom protocol on top of WebSockets, which means plain WebSocket clients cannot connect without the Socket.io client library. Its biggest operational concern at scale is that socket state is in-process ‚Äî multi-instance deployments need a Redis adapter to share state across nodes.

#### Code Example
```javascript
const http = require('http');
const { Server } = require('socket.io');

const server = http.createServer();
const io = new Server(server, {
  cors: { origin: 'https://myapp.com' },
});

io.on('connection', (socket) => {
  console.log(`Client connected: ${socket.id}`);

  // Join a room for scoped messaging
  socket.on('joinRoom', (room) => {
    socket.join(room);
    io.to(room).emit('notification', `${socket.id} joined ${room}`);
  });

  // Broadcast only to a specific room
  socket.on('message', ({ room, text }) => {
    io.to(room).emit('message', { from: socket.id, text });
  });

  socket.on('disconnect', (reason) => {
    console.log(`Disconnected: ${socket.id} ‚Äî reason: ${reason}`);
  });
});

server.listen(3000);
```

#### Do's and Don'ts

**Do's**
- Use rooms to scope messages to groups of clients rather than broadcasting to all connected sockets.
- Always handle the `disconnect` event to clean up any state tied to that socket.
- Use namespaces to separate concerns (e.g., `/chat`, `/notifications`) in larger applications.

**Don'ts**
- Don't store application-level socket state in-process when running multiple server instances ‚Äî use the Socket.io Redis adapter.
- Avoid sending large payloads over Socket.io ‚Äî it is optimised for small, frequent messages.
- Don't use Socket.io when you need plain WebSocket clients to connect; use the `ws` library instead.

#### Common Follow-up Questions
- How do you scale Socket.io across multiple server instances?
- What is the difference between `socket.emit`, `io.emit`, and `io.to(room).emit`?
- When would you choose raw WebSockets (`ws`) over Socket.io?

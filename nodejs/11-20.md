### **11. Managing Dependencies in Node.js**

> **Level:** Beginner

#### Why This Is Asked
A baseline expectation — any Node.js developer is expected to understand how dependencies are categorised, installed, and audited before merging code.

#### Explanation
Dependencies are declared in `package.json` under three keys: `dependencies` (required at runtime), `devDependencies` (build tools, test runners — not shipped to production), and `peerDependencies` (packages the consumer must provide, common in libraries and plugins). `npm install` installs both `dependencies` and `devDependencies`; `npm install --omit=dev` (formerly `--production`) installs only runtime dependencies.

#### Code Example
```bash
# Add a runtime dependency
npm install express

# Add dev-only tooling
npm install --save-dev jest nodemon

# Check for known vulnerabilities
npm audit

# See which packages have newer versions
npm outdated

# Update all packages within allowed ranges
npm update
```

```json
{
  "dependencies": { "express": "^4.18.2" },
  "devDependencies": { "jest": "^29.0.0", "nodemon": "^3.0.0" }
}
```

#### Do's and Don'ts

**Do's**
- Run `npm audit` regularly and resolve high/critical vulnerabilities before releasing.
- Put only what is genuinely needed at runtime into `dependencies` — keep production installs lean.
- Review a package's weekly downloads, maintenance status, and license before adding it.

**Don'ts**
- Don't add runtime code to `devDependencies` — it won't be installed in production, causing crashes.
- Avoid committing `node_modules` — commit `package-lock.json` instead and let CI install from it.

#### Common Follow-up Questions
- What is the difference between `^` (caret) and `~` (tilde) version ranges in `package.json`?
- What does `npm ci` do differently from `npm install`?
- Why would a library declare a dependency as `peerDependency` instead of `dependency`?

### **12. Middleware Stack in Express.js**

> **Level:** Intermediate

#### Why This Is Asked
A deeper follow-up to Q1 — tests whether the candidate understands middleware composition, can debug ordering issues, and knows how to scope middleware correctly.

#### Explanation
Express builds an internal stack of middleware functions and route handlers. When a request arrives, Express walks the stack top to bottom, calling each entry whose path pattern matches the request URL. Calling `next()` advances to the next match. If no middleware sends a response, Express returns a default 404. Error-handling middleware `(err, req, res, next)` only receives control when `next(err)` is called with an argument.

#### Code Example
```javascript
const express = require('express');
const app = express();

// 1. Body parser — must come before any route that reads req.body
app.use(express.json());

// 2. Global request logger
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next();
});

// 3. Auth guard scoped to /api only
app.use('/api', (req, res, next) => {
  if (!req.headers.authorization) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  next();
});

// 4. Route handler
app.get('/api/users', (req, res) => {
  res.json([{ id: 1, name: 'Alice' }]);
});

// 5. Error handler — must be last, must have exactly 4 arguments
app.use((err, req, res, next) => {
  res.status(500).json({ error: err.message });
});

app.listen(3000);
```

#### Do's and Don'ts

**Do's**
- Place body-parsing middleware before any route that reads `req.body`.
- Scope auth and other conditional middleware to specific path prefixes to avoid overhead on public routes.
- Always add a 4-argument error handler as the last entry in the stack.

**Don'ts**
- Don't call `next()` after `res.send()` or `res.json()` — the response is already sent.
- Avoid placing heavy, slow middleware globally when it only applies to a subset of routes.

#### Common Follow-up Questions
- How does Express determine which middleware runs for a given request?
- How do you pass data from one middleware to the next within the same request?
- What is the difference between `app.use('/path', fn)` and `app.get('/path', fn)`?

### **13. Load Testing a Node.js Application**

> **Level:** Advanced

#### Why This Is Asked
Senior candidates are expected to know how to validate performance before it hits production — this tests familiarity with tooling, metrics, and how to interpret results.

#### Explanation
Load testing simulates concurrent users to measure throughput (requests/sec), latency (p50, p95, p99), and error rate under stress. Common tools: **Artillery** (YAML config, easy CI integration), **k6** (JavaScript scripting, great for complex flows), **autocannon** (Node.js native, fast single-endpoint benchmarks). The goal is finding the saturation point — where latency climbs and errors begin. Always test against a staging environment that mirrors production.

#### Code Example
```bash
# Quick benchmark with autocannon
npx autocannon -c 100 -d 30 http://localhost:3000/api/users
# -c 100 : 100 concurrent connections
# -d 30  : run for 30 seconds — observe p95/p99 in the output

# Artillery YAML config (save as load-test.yml)
# config:
#   target: "http://localhost:3000"
#   phases:
#     - duration: 60
#       arrivalRate: 10    # ramp from 10 users/sec
#     - duration: 120
#       arrivalRate: 100   # sustain 100 users/sec
# scenarios:
#   - flow:
#       - get:
#           url: "/api/users"

artillery run load-test.yml --output results.json
artillery report results.json
```

#### Do's and Don'ts

**Do's**
- Watch p95 and p99 latency — averages hide the tail experience that real users feel.
- Run load tests as a CI step to catch regressions before they reach production.
- Combine load testing with profiling (`clinic.js flame`) to identify what is actually slow.
- Include authentication, error paths, and timeout scenarios — not just the happy path.

**Don'ts**
- Never run a full load test against a shared production database — it can cause an outage.
- Don't treat a passing average latency as a green signal — check error rate and tail percentiles.
- Avoid testing in isolation; include realistic payloads and think about downstream service latency.

#### Common Follow-up Questions
- What is the difference between load testing, stress testing, and soak testing?
- How would you interpret a p99 that is 10× the p50 latency?
- What would you change if load testing revealed the database was the bottleneck?

### **14. Advantages of Using Node.js**

> **Level:** Beginner

#### Why This Is Asked
Tests whether the candidate can articulate *when* Node.js is the right tool — not just "it's fast," but specifically why and for which workloads.

#### Explanation
Node.js excels at **I/O-bound, high-concurrency workloads** — REST APIs, API gateways, real-time servers, microservices, and BFF (backend-for-frontend) layers — because its non-blocking event loop handles thousands of concurrent connections without the cost of one OS thread per connection. The unified language (JavaScript) across frontend and backend, and the enormous npm ecosystem, are additional practical advantages. Node.js is a poor fit for **CPU-intensive work** (video encoding, ML inference) on the main thread.

#### Code Example
```javascript
// Node.js can handle thousands of concurrent connections
// because awaiting I/O yields the thread to serve other requests
const http = require('http');

http.createServer(async (req, res) => {
  // While this query is in flight, the event loop serves other requests
  const data = await database.query('SELECT * FROM users');
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify(data));
}).listen(3000);
```

#### Do's and Don'ts

**Do's**
- Use Node.js for real-time apps (chat, live dashboards), REST/GraphQL APIs, and streaming pipelines.
- Add TypeScript for larger codebases — the Node.js ecosystem has excellent TS support.
- Evaluate npm packages for maintenance status, security advisories, and bundle size before adding them.

**Don'ts**
- Don't run heavy CPU-bound tasks on the main thread — use `worker_threads` or a separate service.
- Avoid over-engineering with Node.js for use cases where a simpler tool (serverless function, edge worker) would suffice.

#### Common Follow-up Questions
- In what scenarios would you choose Go or Python over Node.js for a backend service?
- How does Node.js achieve concurrency if it's single-threaded?
- What is the difference between Node.js's concurrency model and Go's goroutines?

### **15. Implementing OAuth Authentication**

> **Level:** Advanced

#### Why This Is Asked
OAuth is used in nearly every production app that integrates with third-party services — this tests practical security knowledge, not just familiarity with a library's API.

#### Explanation
OAuth 2.0 is an **authorisation** framework, not an authentication protocol. It lets a user grant a third-party app access to their resources on another service without sharing credentials. The most common server-side flow is **Authorization Code**: user is redirected to the provider → user approves → provider redirects back with a short-lived code → server exchanges the code for access/refresh tokens. OpenID Connect (OIDC) builds on top of OAuth 2.0 to add identity (authentication).

#### Code Example
```javascript
const express = require('express');
const passport = require('passport');
const GoogleStrategy = require('passport-google-oauth20').Strategy;
const session = require('express-session');

const app = express();

app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
}));
app.use(passport.initialize());
app.use(passport.session());

passport.use(new GoogleStrategy({
  clientID: process.env.GOOGLE_CLIENT_ID,
  clientSecret: process.env.GOOGLE_CLIENT_SECRET,
  callbackURL: '/auth/google/callback',
}, async (accessToken, refreshToken, profile, done) => {
  // Find or create the user in your own database
  const user = await User.findOrCreate({ googleId: profile.id });
  return done(null, user);
}));

passport.serializeUser((user, done) => done(null, user.id));
passport.deserializeUser((id, done) => User.findById(id, done));

app.get('/auth/google', passport.authenticate('google', { scope: ['profile', 'email'] }));

app.get('/auth/google/callback',
  passport.authenticate('google', { failureRedirect: '/login' }),
  (req, res) => res.redirect('/dashboard')
);
```

#### Do's and Don'ts

**Do's**
- Store `CLIENT_SECRET` in environment variables — never commit it to version control.
- Always use HTTPS for OAuth callback URLs.
- Validate the `state` parameter during the callback to prevent CSRF attacks.
- Use short-lived access tokens and store refresh tokens securely (httpOnly, Secure cookies).

**Don'ts**
- Don't store access tokens in `localStorage` — they are vulnerable to XSS theft.
- Avoid requesting OAuth scopes beyond what your application actually needs.
- Don't implement the OAuth flow from scratch — use Passport.js or a managed auth provider (Auth0, Clerk).

#### Common Follow-up Questions
- What is the difference between OAuth 2.0 and OpenID Connect (OIDC)?
- What is PKCE and when is it required?
- How would you handle access token expiry and refresh in a long-lived production session?

### **16. Purpose of the tty Module**

> **Level:** Intermediate

#### Why This Is Asked
Appears in interviews to test depth of Node.js built-in knowledge — less commonly used, but reveals whether a candidate has explored the ecosystem beyond web frameworks.

#### Explanation
The `tty` module provides utilities for working with TTY (teletypewriter / terminal) streams. Its primary use is detecting whether a process is attached to an interactive terminal vs. piped to another process (e.g., `node app.js | grep error`). CLI tools use this to conditionally enable colour output, progress spinners, and interactive prompts only when running in a real terminal. Internally, `readline` and `console` use `tty`.

#### Code Example
```javascript
const tty = require('tty');

// Guard coloured output — disabled when piped to grep/less
if (tty.isatty(process.stdout.fd)) {
  console.log('\x1b[32m✓ Running in a terminal — colours enabled\x1b[0m');
} else {
  console.log('Running in non-interactive mode — plain output');
}

// Read terminal dimensions for layout decisions
if (process.stdout.isTTY) {
  const { columns, rows } = process.stdout;
  console.log(`Terminal: ${columns}×${rows}`);
}
```

#### Do's and Don'ts

**Do's**
- Use `tty.isatty()` or `process.stdout.isTTY` to guard any interactive terminal features.
- Respect the `NO_COLOR` environment variable — some users or CI systems need plain output.

**Don'ts**
- Don't use `tty` directly for high-level terminal interaction — use `readline`, `inquirer`, or `@clack/prompts` instead.
- Don't assume `process.stdout` is always a TTY — CI pipelines and shell pipes are not terminals.

#### Common Follow-up Questions
- How would you build a CLI tool that automatically disables colour when its output is piped?
- What additional functionality does `readline` provide on top of `tty`?

### **17. Performing HTTP Requests in Node.js**

> **Level:** Beginner

#### Why This Is Asked
Every Node.js backend makes outgoing HTTP calls — tests whether the candidate knows the built-in vs. library options and can handle errors and timeouts correctly.

#### Explanation
Node.js includes `http` and `https` built-in modules, but they are verbose. The built-in `fetch` API (available globally since Node.js 18) is the modern zero-dependency choice. For older Node.js versions or richer configuration, `axios` is the most widely used library. For streaming large responses, the built-in `https` module or `got` (stream-friendly) are better fits than `axios`.

#### Code Example
```javascript
// Built-in fetch — Node.js 18+ (zero dependencies, preferred for new code)
const response = await fetch('https://api.example.com/users', {
  headers: { Authorization: `Bearer ${token}` },
  signal: AbortSignal.timeout(5000), // 5 s timeout
});
if (!response.ok) throw new Error(`HTTP ${response.status}`);
const users = await response.json();

// axios — works in Node.js 14+ (rich config, interceptors)
const axios = require('axios');
const { data } = await axios.get('https://api.example.com/users', {
  timeout: 5000,
  headers: { Authorization: `Bearer ${token}` },
});

// Built-in https — shown for reference (avoid in application code)
const https = require('https');
https.get('https://api.example.com/users', (res) => {
  let body = '';
  res.on('data', (chunk) => body += chunk);
  res.on('end', () => console.log(JSON.parse(body)));
  res.on('error', console.error);
});
```

#### Do's and Don'ts

**Do's**
- Always set a request timeout — a hanging request will eventually exhaust your connection pool.
- Check the HTTP status code before parsing the body.
- Implement retries with exponential backoff for transient errors (5xx, network failures).

**Don'ts**
- Don't accumulate large response bodies in memory — stream them to the client or a file.
- Avoid the raw `https` module in application code — the ergonomics are poor and error handling is easy to miss.

#### Common Follow-up Questions
- How would you retry a failed HTTP request with exponential backoff and jitter?
- What is the difference between a connection timeout and a response timeout?
- How would you implement a circuit breaker for outgoing HTTP calls?

### **18. Purpose of the Crypto Module**

> **Level:** Intermediate

#### Why This Is Asked
Security-sensitive operations appear in almost every production backend — interviewers want to confirm the candidate uses cryptographic primitives correctly rather than rolling their own or misusing them.

#### Explanation
The `crypto` module provides cryptographic primitives: one-way hashing (SHA-256, SHA-512), HMAC for signing, symmetric encryption (AES-256-GCM), asymmetric operations (RSA, ECDSA), and secure random bytes. One important distinction: `crypto` is **not** suitable for password storage — use `bcrypt` or `argon2`, which are purpose-built with configurable work factors that resist brute-force attacks.

#### Code Example
```javascript
const crypto = require('crypto');

// One-way hash — use for data integrity, NOT passwords
const hash = crypto.createHash('sha256').update('some data').digest('hex');

// HMAC — sign data to detect tampering (e.g., JWT-style tokens, webhooks)
const hmac = crypto
  .createHmac('sha256', process.env.SECRET_KEY)
  .update('payload')
  .digest('hex');

// Cryptographically secure random token (session IDs, reset links, API keys)
const token = crypto.randomBytes(32).toString('hex');

// Constant-time comparison — prevents timing attacks when comparing secrets
const isValid = crypto.timingSafeEqual(
  Buffer.from(providedToken),
  Buffer.from(storedToken)
);
```

#### Do's and Don'ts

**Do's**
- Use `crypto.randomBytes()` for all security tokens — `Math.random()` is not cryptographically secure.
- Use `crypto.timingSafeEqual()` when comparing secrets to prevent timing side-channel attacks.
- Use AES-256-GCM for symmetric encryption — it provides authenticated encryption (detects tampering).
- Use `bcrypt` or `argon2` for password hashing, not SHA-256.

**Don'ts**
- Never use MD5 or SHA-1 for security-sensitive hashing — they are considered broken.
- Don't reuse an IV (initialisation vector) across encryption operations — generate a fresh one each time.
- Avoid storing plaintext passwords or reversibly encrypted passwords — always use a one-way hash with a work factor.

#### Common Follow-up Questions
- Why is `bcrypt` better than SHA-256 for password storage?
- What is the difference between hashing, encryption, and encoding?
- What is a timing attack and how does `timingSafeEqual` prevent it?

### **19. Package-lock.json in Node.js**

> **Level:** Beginner

#### Why This Is Asked
Failing to understand lock files leads to "works on my machine" bugs and introduces supply-chain security risks — this is a foundational topic for any developer working in a team.

#### Explanation
`package-lock.json` is auto-generated by npm and records the exact version, registry URL, and integrity hash of every installed package, including all transitive dependencies. This guarantees that `npm ci` installs the exact same dependency tree on every machine and in every CI run. Without a lock file, two developers running `npm install` at different times may silently get different transitive dependency versions.

#### Code Example
```json
{
  "name": "my-app",
  "lockfileVersion": 3,
  "packages": {
    "node_modules/express": {
      "version": "4.18.2",
      "resolved": "https://registry.npmjs.org/express/-/express-4.18.2.tgz",
      "integrity": "sha512-abc123..."
    }
  }
}
```

```bash
# In CI/CD — use npm ci, not npm install
# npm ci: reads the lock file exactly, errors if package.json is out of sync
npm ci

# In development — use npm install (updates the lock file as needed)
npm install
```

#### Do's and Don'ts

**Do's**
- Always commit `package-lock.json` — it is what makes builds reproducible across machines.
- Use `npm ci` in CI/CD pipelines — it is faster and guarantees lock-file fidelity.
- Review `package-lock.json` diffs in pull requests — unexpected changes can signal dependency confusion or supply-chain tampering.

**Don'ts**
- Don't manually edit `package-lock.json` — let npm manage it.
- Don't delete and regenerate the lock file casually — it silently upgrades all transitive dependencies.

#### Common Follow-up Questions
- What is the practical difference between `npm install` and `npm ci`?
- What changed between `lockfileVersion` 1, 2, and 3?
- How does Yarn's `yarn.lock` differ from `package-lock.json`?

### **20. Securing a Node.js Application**

> **Level:** Advanced

#### Why This Is Asked
Security is a shared responsibility — senior Node.js developers are expected to know common attack vectors and their mitigations, not rely entirely on a dedicated security team to catch issues.

#### Explanation
Node.js applications face the same threats as all web backends: injection attacks (SQL, NoSQL, command), XSS, CSRF, broken authentication, insecure dependencies, and sensitive data exposure. Defence is layered: secure HTTP headers, strict input validation, rate limiting, dependency auditing, least-privilege DB credentials, and no secrets in code or logs.

#### Code Example
```javascript
const express = require('express');
const helmet = require('helmet');
const rateLimit = require('express-rate-limit');
const mongoSanitize = require('express-mongo-sanitize');

const app = express();

// Secure HTTP headers: X-Frame-Options, CSP, HSTS, X-Content-Type-Options, etc.
app.use(helmet());

// Rate limit all API routes — prevent brute-force and DoS
app.use('/api/', rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100,
  standardHeaders: true,
}));

// Limit body size to prevent payload-based DoS
app.use(express.json({ limit: '10kb' }));

// Sanitise MongoDB query operators in user input
app.use(mongoSanitize());

// Never expose internal error details to clients in production
app.use((err, req, res, next) => {
  const status = err.statusCode || 500;
  res.status(status).json({
    error: process.env.NODE_ENV === 'production'
      ? 'Internal Server Error'
      : err.message,
  });
});

app.listen(3000);
```

#### Do's and Don'ts

**Do's**
- Use `helmet` on every Express application — it closes common HTTP header vulnerabilities with one line.
- Validate and sanitise all user input at the system boundary using `zod` or `joi`.
- Store all secrets in environment variables — never hardcode them or log them.
- Run `npm audit --audit-level=high` in CI and block deployments on critical findings.
- Apply principle of least privilege: DB users should have only the permissions they need.

**Don'ts**
- Don't trust user-provided data for file paths, shell commands, or database queries.
- Avoid returning stack traces or internal error messages to clients in production.
- Don't use `eval()` or `new Function()` with user input — it enables arbitrary code execution.
- Never run Node.js processes as `root` in production.

#### Common Follow-up Questions
- What specific HTTP headers does `helmet` set and what does each one protect against?
- How would you prevent HTTP parameter pollution attacks?
- What is the difference between authentication and authorisation, and how do you implement both in an Express app?

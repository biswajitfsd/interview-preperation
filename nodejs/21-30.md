### **21. Performing File Operations in Node.js**

> **Level:** Beginner

#### Why This Is Asked
File I/O is fundamental to backend development — tests whether the candidate knows how to perform common operations correctly and asynchronously.

#### Explanation
Node.js provides the `fs` module for file operations: reading, writing, appending, deleting, and watching files. Every operation has both a callback-based async version and a synchronous version. Since Node.js v10, the `fs/promises` API provides Promise-based equivalents, making them compatible with `async/await` without wrapping in `util.promisify`.

#### Code Example
```javascript
const fs = require('fs/promises');

// Write a file (creates or overwrites)
await fs.writeFile('example.txt', 'Hello, World!', 'utf8');

// Read a file
const content = await fs.readFile('example.txt', 'utf8');
console.log(content); // Hello, World!

// Append to a file
await fs.appendFile('example.txt', '\nAppended line');

// Delete a file
await fs.unlink('example.txt');

// Check if a file exists (use try/catch instead of fs.exists — deprecated)
try {
  await fs.access('example.txt');
  console.log('File exists');
} catch {
  console.log('File does not exist');
}
```

#### Do's and Don'ts

**Do's**
- Use the `fs/promises` API with `async/await` for clean, non-blocking file operations in application code.
- Always handle errors — file not found, permission denied, and disk full are all real-world failures.
- Use streams (`fs.createReadStream`) for large files instead of reading the entire file into memory.

**Don'ts**
- Avoid `fs.readFileSync` and other synchronous methods in server code — they block the event loop for all concurrent requests.
- Don't use `fs.exists()` — it is deprecated. Use `fs.access()` in a try/catch instead.

#### Common Follow-up Questions
- When would `fs.readFileSync` be acceptable to use?
- How would you read a 10 GB log file without running out of memory?
- What is the difference between `fs.writeFile` and `fs.createWriteStream`?

### **22. Difference Between fs.readFile() and fs.readFileSync()**

> **Level:** Beginner

#### Why This Is Asked
A targeted question about blocking vs. non-blocking I/O — the distinction is fundamental to understanding why Node.js is designed the way it is.

#### Explanation
`fs.readFile()` is asynchronous and non-blocking: it hands the I/O off to libuv, returns immediately, and invokes the callback (or resolves the Promise) when the data is ready. The event loop continues serving other requests in the meantime. `fs.readFileSync()` is synchronous and blocking: it halts the entire Node.js thread until the file is fully read — blocking every other incoming request for that duration.

#### Code Example
```javascript
const fs = require('fs');
const fsPromises = require('fs/promises');

// Callback-style async (non-blocking)
fs.readFile('example.txt', 'utf8', (err, data) => {
  if (err) throw err;
  console.log('Async (callback):', data);
});

// Promise-style async (non-blocking, preferred)
const data = await fsPromises.readFile('example.txt', 'utf8');
console.log('Async (await):', data);

// Synchronous (BLOCKS the thread until complete)
const syncData = fs.readFileSync('example.txt', 'utf8');
console.log('Sync:', syncData);

console.log('This runs before async results above');
```

#### Do's and Don'ts

**Do's**
- Use the async API (`fs/promises`) in all server and library code.
- Use `fs.readFileSync` only in CLI scripts, one-off migration scripts, or module initialisation that must complete before the server starts.

**Don'ts**
- Don't use `readFileSync` inside request handlers, middleware, or anywhere the event loop must stay responsive.
- Avoid reading large files into memory with either method — use `createReadStream` instead.

#### Common Follow-up Questions
- What happens to other incoming requests while `readFileSync` is executing?
- How does `util.promisify` relate to the callback-based `fs` API?
- Why does Node.js provide both sync and async versions if sync is generally harmful in a server?

### **23. Role of Child_Process Module in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Tests whether the candidate knows how to break out of Node.js's single-threaded model when needed — running shell commands, spawning services, or offloading CPU work.

#### Explanation
The `child_process` module lets Node.js spawn external processes. It provides four methods: `exec` (shell command, buffers output — simple, limited to small output), `execFile` (like `exec` but no shell — slightly safer), `spawn` (streaming I/O — best for large output or long-running processes), and `fork` (spawns a new Node.js process with an IPC channel for message passing). Use `spawn` or `fork` for production code; `exec` is fine for scripting.

#### Code Example
```javascript
const { exec, spawn, fork } = require('child_process');

// exec: simple shell commands with buffered output
exec('ls -la', (error, stdout, stderr) => {
  if (error) { console.error('Error:', error.message); return; }
  console.log(stdout);
});

// spawn: streaming output — good for large output or long-running processes
const child = spawn('find', ['.', '-name', '*.js']);
child.stdout.on('data', (data) => process.stdout.write(data));
child.stderr.on('data', (err) => process.stderr.write(err));
child.on('close', (code) => console.log(`Exited with code ${code}`));

// fork: spawn a Node.js child with IPC (message passing)
const worker = fork('./worker.js');
worker.send({ task: 'processData', payload: [1, 2, 3] });
worker.on('message', (result) => console.log('Result:', result));
```

#### Do's and Don'ts

**Do's**
- Use `spawn` over `exec` for commands that produce large output — `exec` buffers everything in memory.
- Use `fork` for CPU-intensive Node.js work so the parent's event loop stays free.
- Always handle the `'error'` and `'close'` events on child processes.

**Don'ts**
- Never pass unsanitised user input to `exec` — it runs through a shell and enables command injection.
- Avoid leaving child processes running after the parent exits — clean up with `child.kill()` or `process.on('exit')`.

#### Common Follow-up Questions
- What is the difference between `exec`, `spawn`, and `fork`?
- How does `fork` communicate between parent and child processes?
- When would you use `worker_threads` instead of `child_process`?

### **24. Role of the REPL in Node.js**

> **Level:** Beginner

#### Why This Is Asked
Tests basic familiarity with Node.js development tooling and the ability to quickly prototype and debug.

#### Explanation
REPL stands for Read-Eval-Print-Loop. It is Node.js's interactive shell — type JavaScript, press Enter, see the result immediately. It is useful for quickly testing an API, inspecting an object, or experimenting with a built-in module without writing a full script. You can also create a custom REPL programmatically to embed an interactive shell in your own CLI tool.

#### Code Example
```bash
# Start the Node.js REPL
node

# Inside the REPL:
> const arr = [1, 2, 3, 4, 5]
undefined
> arr.filter(n => n % 2 === 0)
[ 2, 4 ]
> arr.reduce((sum, n) => sum + n, 0)
15
> .help          # list REPL commands
> .exit          # or Ctrl+D to quit
```

```javascript
// Custom REPL — embed an interactive shell in your CLI tool
const repl = require('repl');

const server = repl.start({ prompt: 'myapp> ' });

// Expose application context to the REPL
server.context.db = myDatabase;
server.context.config = appConfig;
```

#### Do's and Don'ts

**Do's**
- Use the REPL to quickly test unfamiliar APIs, prototype small functions, or inspect module behaviour.
- Use `.load <file>` inside the REPL to load and execute a script in the REPL context.
- Use `.save <file>` to export your REPL session to a file.

**Don'ts**
- Don't use the REPL as a substitute for automated tests — REPL sessions are not reproducible.
- Avoid running database mutations in the REPL against production — it has no undo.

#### Common Follow-up Questions
- How would you pre-load modules or variables into a custom REPL?
- What is the difference between REPL and running `node script.js`?
- How does the REPL handle multi-line expressions?

### **25. Managing Sessions in Express.js**

> **Level:** Intermediate

#### Why This Is Asked
Session management is core to authentication — tests whether the candidate knows secure session configuration and the operational implications of different session stores.

#### Explanation
HTTP is stateless — sessions add statefulness by storing a session ID in a cookie and associating it with server-side data. `express-session` is the standard library. By default it stores session data in memory, which is lost on server restart and doesn't work across multiple instances. Production deployments must use a persistent store — Redis (via `connect-redis`) is the most common choice.

#### Code Example
```javascript
const express = require('express');
const session = require('express-session');
const RedisStore = require('connect-redis').default;
const { createClient } = require('redis');

const app = express();

const redisClient = createClient({ url: process.env.REDIS_URL });
await redisClient.connect();

app.use(session({
  store: new RedisStore({ client: redisClient }),
  secret: process.env.SESSION_SECRET,   // used to sign the session cookie
  resave: false,                         // don't save unmodified sessions
  saveUninitialized: false,              // don't create sessions for unauthenticated requests
  cookie: {
    httpOnly: true,   // prevent JS access to the cookie
    secure: true,     // send only over HTTPS
    maxAge: 24 * 60 * 60 * 1000, // 24 hours
    sameSite: 'strict',
  },
}));

app.post('/login', async (req, res) => {
  const user = await authenticate(req.body);
  req.session.userId = user.id;  // session data is serialised to Redis
  res.json({ message: 'Logged in' });
});

app.get('/me', (req, res) => {
  if (!req.session.userId) return res.status(401).json({ error: 'Not authenticated' });
  res.json({ userId: req.session.userId });
});
```

#### Do's and Don'ts

**Do's**
- Use a persistent session store (Redis, database) in production — the default in-memory store is not suitable for production.
- Set `httpOnly: true` and `secure: true` on the session cookie to prevent XSS and MITM theft.
- Regenerate the session ID after a successful login to prevent session fixation attacks (`req.session.regenerate()`).

**Don'ts**
- Don't store sensitive data (passwords, payment info) directly in the session — store only a minimal identifier.
- Avoid hardcoding the session `secret` — load it from an environment variable or a secrets manager.

#### Common Follow-up Questions
- What is a session fixation attack and how does regenerating the session ID prevent it?
- What is the difference between sessions and JWTs for maintaining authentication state?
- What are the operational trade-offs between Redis sessions and stateless JWTs in a multi-instance deployment?

### **26. Role of Cluster Module in Node.js**

> **Level:** Advanced

#### Why This Is Asked
Tests understanding of how to scale a Node.js application across CPU cores — and whether the candidate knows the difference between clustering and worker threads.

#### Explanation
Node.js runs on a single thread, so a single process cannot use more than one CPU core. The `cluster` module creates multiple child processes (workers) that all share the same server port. The OS distributes incoming connections across workers in a round-robin fashion. Each worker is a full Node.js process with its own event loop and memory — they don't share state. For shared state across workers, use an external store like Redis.

#### Code Example
```javascript
const cluster = require('cluster');
const http = require('http');
const os = require('os');

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  console.log(`Primary ${process.pid} — forking ${numCPUs} workers`);

  for (let i = 0; i < numCPUs; i++) cluster.fork();

  // Restart any worker that crashes
  cluster.on('exit', (worker, code, signal) => {
    console.warn(`Worker ${worker.process.pid} died (${signal || code}) — restarting`);
    cluster.fork();
  });
} else {
  // Each worker runs its own HTTP server on the same port
  http.createServer((req, res) => {
    res.end(`Handled by worker ${process.pid}`);
  }).listen(3000);

  console.log(`Worker ${process.pid} started`);
}
```

#### Do's and Don'ts

**Do's**
- Always restart crashed workers in the `'exit'` event handler.
- Use an external store (Redis, database) for any state that must be shared across workers.
- Use a process manager like PM2 (`pm2 start app.js -i max`) instead of managing clustering manually in application code.

**Don'ts**
- Don't store session state, caches, or socket connections in-process when using cluster — they will be silently isolated per worker.
- Avoid using `cluster` for CPU-bound work within a single request — use `worker_threads` instead.

#### Common Follow-up Questions
- What is the difference between `cluster` and `worker_threads`?
- How does the OS distribute connections across cluster workers?
- What problem does PM2 solve over writing your own cluster logic?

### **27. Handling Multiple Requests in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Tests whether the candidate can articulate Node.js's concurrency model clearly — many candidates confuse "single-threaded" with "can only handle one request at a time."

#### Explanation
Node.js handles many concurrent requests on a single thread by never blocking that thread on I/O. When a request triggers an I/O operation (database query, file read, HTTP call), Node.js registers a callback and immediately returns to the event loop to handle the next request. When the I/O completes, the callback is queued and eventually executed. This is why non-blocking async code is critical — any synchronous block stalls every concurrent request.

#### Code Example
```javascript
const http = require('http');

// Simulation: this server handles 1000 concurrent requests,
// each waiting 1 second for "I/O", yet total time ≈ 1 second — not 1000 seconds.
http.createServer(async (req, res) => {
  // Simulated async I/O (database call, external API, etc.)
  await new Promise((resolve) => setTimeout(resolve, 1000));
  res.end(`Request to ${req.url} handled by PID ${process.pid}`);
}).listen(3000, () => {
  console.log('Server running — try: npx autocannon -c 1000 http://localhost:3000');
});
```

#### Do's and Don'ts

**Do's**
- Write async code for all I/O operations — it is the key to Node.js's concurrency.
- Use connection pooling for database clients so the pool manages concurrent DB connections efficiently.
- Add request timeouts to prevent slow clients from holding connections open indefinitely.

**Don'ts**
- Don't run CPU-intensive synchronous code in request handlers — a 500 ms CPU task blocks all 1000 concurrent requests for 500 ms.
- Avoid unbounded concurrency — if every request fans out to 50 DB queries, 1000 concurrent requests become 50,000 simultaneous queries. Use queues or rate-limit fanout.

#### Common Follow-up Questions
- How does Node.js serve 10,000 concurrent connections on a single thread?
- What happens to all queued requests if a single synchronous operation blocks for 2 seconds?
- When would you add a request queue or back-pressure mechanism?

### **28. Purpose of Buffer Class in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Binary data handling appears when working with streams, file I/O, network protocols, and cryptography — tests whether the candidate is comfortable outside the JavaScript string world.

#### Explanation
JavaScript strings are Unicode — they cannot represent arbitrary binary data without corruption. `Buffer` is Node.js's solution: a fixed-size allocation of raw memory that stores bytes. Buffers are instances of `Uint8Array` and integrate naturally with streams and the `fs`/`crypto`/`net` modules. You should always specify an encoding when converting between strings and Buffers, and prefer `Buffer.from()` over `new Buffer()` (deprecated since Node.js 6 due to security issues).

#### Code Example
```javascript
// Create a Buffer from a string
const buf = Buffer.from('Hello, World!', 'utf8');
console.log(buf);           // <Buffer 48 65 6c 6c 6f 2c 20 57 6f 72 6c 64 21>
console.log(buf.length);    // 13 (bytes, not characters)
console.log(buf.toString('utf8'));  // 'Hello, World!'
console.log(buf.toString('base64')); // 'SGVsbG8sIFdvcmxkIQ=='

// Allocate a buffer of a fixed size (filled with zeros)
const zeroed = Buffer.alloc(16);

// Allocate without zeroing — faster but contains old memory (unsafe for user-facing output)
const unsafe = Buffer.allocUnsafe(16);

// Concatenate buffers
const a = Buffer.from('Hello');
const b = Buffer.from(' World');
const combined = Buffer.concat([a, b]);
console.log(combined.toString()); // 'Hello World'
```

#### Do's and Don'ts

**Do's**
- Always specify an encoding when calling `buf.toString()` — defaults to `'utf8'` but being explicit prevents bugs.
- Use `Buffer.alloc()` for security-sensitive operations — it zeroes memory.
- Use `Buffer.concat()` to join buffers — manually slicing and copying is error-prone.

**Don'ts**
- Never use `new Buffer()` — it is deprecated and was removed in Node.js 10 for security reasons.
- Avoid creating very large Buffers in memory — stream the data instead.
- Don't use `Buffer.allocUnsafe()` for buffers that will be sent to users without first filling them.

#### Common Follow-up Questions
- What is the difference between `Buffer.alloc()` and `Buffer.allocUnsafe()`?
- Why does `Buffer.from('€').length` return 3 even though `'€'` is one character?
- How does `Buffer` relate to `Uint8Array`?

### **29. Creating a Child Process in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
A practical follow-up to Q23 — tests whether the candidate can choose the right `child_process` method for the situation and handle IPC correctly.

#### Explanation
`child_process` offers four creation methods with different trade-offs: `exec` (shell, buffered — simple scripting), `execFile` (no shell overhead, buffered), `spawn` (streaming, no shell — for large output or long-running commands), and `fork` (Node.js subprocess with a built-in IPC channel — ideal for CPU-bound tasks). For CPU-bound work where you want shared memory, `worker_threads` is more efficient than `fork` because it avoids process startup overhead.

#### Code Example
```javascript
const { spawn, fork } = require('child_process');

// spawn — stream a long-running process output
const ls = spawn('ls', ['-lh', '/usr']);

ls.stdout.on('data', (data) => process.stdout.write(data));
ls.stderr.on('data', (data) => process.stderr.write(data));
ls.on('close', (code) => console.log(`ls exited with code ${code}`));

// fork — offload CPU work to a Node.js child process
// worker.js: process.on('message', ({n}) => process.send({ result: fibonacci(n) }));
const worker = fork('./worker.js');

worker.send({ n: 40 });

worker.on('message', ({ result }) => {
  console.log(`Fibonacci(40) = ${result}`);
  worker.kill(); // clean up when done
});

worker.on('error', (err) => console.error('Worker error:', err));
worker.on('exit', (code) => console.log(`Worker exited: ${code}`));
```

#### Do's and Don'ts

**Do's**
- Use `spawn` for external commands with large or streaming output.
- Use `fork` to run CPU-heavy Node.js code without blocking the parent's event loop.
- Always listen for `'error'` and `'exit'` events and clean up child processes when they are no longer needed.

**Don'ts**
- Don't pass unsanitised user input to `exec` — it evaluates the string through a shell.
- Avoid creating a new child process per request in a hot path — use a worker pool instead.

#### Common Follow-up Questions
- How would you implement a worker pool using `child_process.fork`?
- When would you choose `worker_threads` over `fork` for CPU-bound tasks?
- How do you pass large amounts of data between a parent and child process efficiently?

### **30. Streams in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Streams are the idiomatic way to handle I/O in Node.js — this question tests whether the candidate understands back-pressure, the four stream types, and when to use streams over buffered operations.

#### Explanation
Streams process data in chunks rather than loading everything into memory at once. Node.js has four stream types: **Readable** (source — `fs.createReadStream`, `http.IncomingMessage`), **Writable** (destination — `fs.createWriteStream`, `http.ServerResponse`), **Duplex** (both — TCP sockets), and **Transform** (modify data as it passes through — `zlib.createGzip`). Back-pressure is the mechanism by which a Writable signals a Readable to pause when its internal buffer is full — handled automatically when you use `.pipe()` or the `pipeline` utility.

#### Code Example
```javascript
const fs = require('fs');
const zlib = require('zlib');
const { pipeline } = require('stream/promises');

// Anti-pattern: reads entire file into memory
const data = fs.readFileSync('huge-file.txt'); // bad for large files
fs.writeFileSync('output.txt', data);

// Correct: stream the file through gzip compression to disk
// Back-pressure is handled automatically by pipeline
await pipeline(
  fs.createReadStream('huge-file.txt'),
  zlib.createGzip(),
  fs.createWriteStream('huge-file.txt.gz')
);
console.log('Compressed successfully');

// Readable stream as an HTTP response (memory-efficient file serving)
const http = require('http');
http.createServer((req, res) => {
  res.setHeader('Content-Encoding', 'gzip');
  pipeline(
    fs.createReadStream('huge-file.txt'),
    zlib.createGzip(),
    res
  ).catch(console.error);
}).listen(3000);
```

#### Do's and Don'ts

**Do's**
- Use `stream/promises`'s `pipeline()` instead of `.pipe()` — it properly handles errors and back-pressure and cleans up all streams on failure.
- Always handle stream errors — an unhandled `'error'` event on a stream crashes the process.
- Use Transform streams to process data incrementally (CSV parsing, encryption, compression).

**Don'ts**
- Don't read an entire large file into memory with `fs.readFile` when you can stream it.
- Avoid chaining `.pipe()` across many streams without error handlers on each — errors in the middle of the chain can be silently swallowed.
- Don't forget that `pipeline` (not `.pipe()`) is the correct tool for production stream chaining.

#### Common Follow-up Questions
- What is back-pressure and what happens if you ignore it?
- What is the difference between `stream.pipe()` and `stream.pipeline()`?
- How would you implement a custom Transform stream to parse CSV data line by line?

### **31. Async Error Handling in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Async error handling is one of the most common sources of bugs in Node.js applications ŌĆö unhandled rejections silently swallow errors or crash processes in newer Node.js versions.

#### Explanation
In async code, errors propagate through Promise chains rather than the call stack. A rejected Promise that has no `.catch()` or `try/catch` becomes an **unhandled rejection** ŌĆö in Node.js v15+ this terminates the process by default. Express route handlers require explicit `try/catch` or a wrapper because async errors thrown inside them do not automatically reach the error-handling middleware. A global `asyncHandler` wrapper eliminates the boilerplate.

#### Code Example
```javascript
// Problem: Express doesn't catch async errors automatically (pre-v5)
app.get('/user/:id', async (req, res) => {
  const user = await db.findById(req.params.id); // if this throws, Express hangs
  res.json(user);
});

// Solution 1: explicit try/catch in every handler
app.get('/user/:id', async (req, res, next) => {
  try {
    const user = await db.findById(req.params.id);
    res.json(user);
  } catch (err) {
    next(err); // forward to error-handling middleware
  }
});

// Solution 2: asyncHandler wrapper ŌĆö eliminates boilerplate
const asyncHandler = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

app.get('/user/:id', asyncHandler(async (req, res) => {
  const user = await db.findById(req.params.id);
  res.json(user);
}));

// Global safety nets ŌĆö log and exit, never swallow silently
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled rejection at:', promise, 'reason:', reason);
  process.exit(1); // let the process manager restart
});

process.on('uncaughtException', (err) => {
  console.error('Uncaught exception:', err);
  process.exit(1);
});
```

#### Do's and Don'ts

**Do's**
- Use an `asyncHandler` wrapper or Express v5 (which handles async errors natively) to avoid repetitive `try/catch` in every route.
- Always call `next(err)` in Express to forward errors to the error-handling middleware.
- Attach `process.on('unhandledRejection')` as a safety net and exit with a non-zero code so the process manager can restart.

**Don'ts**
- Don't leave `.catch()` or `try/catch` off async functions in request handlers ŌĆö the error silently hangs the response.
- Don't swallow errors in empty `catch` blocks ŌĆö at minimum log them, then re-throw or send an error response.
- Avoid using `process.on('uncaughtException')` to "recover" ŌĆö the process state is undefined after an uncaught exception; exit and restart.

#### Common Follow-up Questions
- How does Express v5 change async error handling compared to v4?
- What is the difference between `unhandledRejection` and `uncaughtException`?
- How would you centralise error formatting (status codes, error codes) across all Express routes?

### **32. Environment Variables & Configuration**

> **Level:** Beginner

#### Why This Is Asked
Misconfigured or leaked secrets are a top cause of security incidents ŌĆö this tests whether the candidate knows how to manage configuration safely across environments.

#### Explanation
`process.env` exposes environment variables as strings. The `dotenv` package loads a `.env` file into `process.env` at startup ŌĆö convenient for local development. In production, variables should be injected by the deployment platform (Docker, Kubernetes secrets, Heroku config vars) rather than stored in `.env` files. Validating the shape of `process.env` at startup (using `zod` or `envalid`) catches missing variables immediately rather than as runtime crashes deep in the application.

#### Code Example
```javascript
// .env (never commit this file)
// DATABASE_URL=postgres://user:pass@localhost:5432/mydb
// PORT=3000
// JWT_SECRET=supersecret

// Load .env in development only
if (process.env.NODE_ENV !== 'production') {
  require('dotenv').config();
}

// Validate config at startup ŌĆö fail fast with a clear error message
const { z } = require('zod');

const envSchema = z.object({
  NODE_ENV:     z.enum(['development', 'test', 'production']),
  PORT:         z.string().regex(/^\d+$/).transform(Number).default('3000'),
  DATABASE_URL: z.string().url(),
  JWT_SECRET:   z.string().min(32),
});

// throws at startup if any required variable is missing or invalid
const env = envSchema.parse(process.env);

// Export a typed config object ŌĆö avoids raw process.env access across the codebase
module.exports = {
  port:        env.PORT,
  databaseUrl: env.DATABASE_URL,
  jwtSecret:   env.JWT_SECRET,
  isProd:      env.NODE_ENV === 'production',
};
```

#### Do's and Don'ts

**Do's**
- Validate all required environment variables at startup ŌĆö fail loudly with a clear message rather than at the usage site.
- Export a single typed config module rather than reading `process.env` directly throughout the codebase.
- Add `.env` to `.gitignore` and commit a `.env.example` with placeholder values as documentation.

**Don'ts**
- Never commit `.env` files containing real secrets to version control.
- Don't store secrets in `package.json` scripts or hard-code them in source files.
- Avoid casting `process.env` values manually everywhere ŌĆö centralise the conversion in the config module.

#### Common Follow-up Questions
- How would you rotate a secret (e.g., JWT secret) without downtime?
- What is the difference between `dotenv` and using a secrets manager like AWS Secrets Manager or HashiCorp Vault?
- How do you handle different configuration values per environment (dev, staging, production)?

### **33. Graceful Shutdown in Node.js**

> **Level:** Advanced

#### Why This Is Asked
Abrupt process termination drops in-flight requests and leaves database connections in an inconsistent state ŌĆö graceful shutdown is expected knowledge for any production Node.js engineer.

#### Explanation
When a container orchestrator (Kubernetes, Docker) stops a service it sends `SIGTERM`, waits a grace period, then sends `SIGKILL`. A Node.js process that ignores `SIGTERM` gets killed mid-request. Graceful shutdown means: stop accepting new connections, finish in-flight requests, close database/Redis connections, then exit. The grace period is typically 10ŌĆō30 seconds.

#### Code Example
```javascript
const express = require('express');
const app = express();

app.get('/health', (req, res) => res.json({ status: 'ok' }));

const server = app.listen(3000, () => console.log('Server started'));

// Track in-flight requests
let connections = 0;
app.use((req, res, next) => {
  connections++;
  res.on('finish', () => connections--);
  next();
});

async function shutdown(signal) {
  console.log(`Received ${signal} ŌĆö starting graceful shutdown`);

  // 1. Stop accepting new HTTP connections
  server.close(async () => {
    console.log('HTTP server closed');

    try {
      // 2. Close external resources
      await db.end();
      await redisClient.quit();
      console.log('DB and Redis connections closed');
      process.exit(0);
    } catch (err) {
      console.error('Error during shutdown:', err);
      process.exit(1);
    }
  });

  // 3. Force exit if graceful shutdown takes too long
  setTimeout(() => {
    console.error('Graceful shutdown timed out ŌĆö forcing exit');
    process.exit(1);
  }, 15_000).unref(); // .unref() prevents the timeout from keeping the process alive
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT',  () => shutdown('SIGINT'));  // Ctrl+C in local dev
```

#### Do's and Don'ts

**Do's**
- Always handle `SIGTERM` ŌĆö it is the standard signal for scheduled shutdowns in container environments.
- Set a hard timeout (15ŌĆō30 s) as a fallback so the process does not hang indefinitely if a connection fails to close.
- Close resources in the right order: stop accepting traffic ŌåÆ finish in-flight requests ŌåÆ close DB ŌåÆ exit.

**Don'ts**
- Don't call `process.exit()` directly in application code ŌĆö it skips cleanup handlers.
- Avoid keeping the process alive after `server.close()` with lingering timers or DB connections ŌĆö use `.unref()` on the fallback timeout.

#### Common Follow-up Questions
- What is the difference between `SIGTERM` and `SIGKILL`? Why can't you catch `SIGKILL`?
- How do you handle graceful shutdown when using the `cluster` module?
- What does Kubernetes do if a pod does not terminate within `terminationGracePeriodSeconds`?

### **34. JWT Authentication in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
JWT is used in nearly every modern Node.js API ŌĆö this tests both the implementation knowledge and the security trade-offs compared to session-based auth.

#### Explanation
A JSON Web Token (JWT) is a self-contained, signed token with three Base64-encoded parts: **header** (algorithm), **payload** (claims), and **signature**. The server signs the token with a secret; any server with the same secret can verify it without a database lookup, making JWTs stateless. The trade-off: a JWT cannot be revoked before it expires ŌĆö a compromised token is valid until expiry. Short-lived access tokens (15 min) plus a refresh token stored in an httpOnly cookie is the standard mitigation.

#### Code Example
```javascript
const jwt = require('jsonwebtoken');
const { jwtSecret, jwtRefreshSecret } = require('./config');

// Sign an access token (short-lived)
function signAccessToken(userId, role) {
  return jwt.sign(
    { sub: userId, role },
    jwtSecret,
    { expiresIn: '15m', issuer: 'myapp' }
  );
}

// Sign a refresh token (long-lived, stored in httpOnly cookie)
function signRefreshToken(userId) {
  return jwt.sign({ sub: userId }, jwtRefreshSecret, { expiresIn: '7d' });
}

// Verify middleware
function authenticate(req, res, next) {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing token' });
  }
  const token = authHeader.slice(7);
  try {
    req.user = jwt.verify(token, jwtSecret, { issuer: 'myapp' });
    next();
  } catch (err) {
    const message = err.name === 'TokenExpiredError' ? 'Token expired' : 'Invalid token';
    res.status(401).json({ error: message });
  }
}

// Refresh endpoint
app.post('/auth/refresh', (req, res) => {
  const refreshToken = req.cookies.refreshToken;
  if (!refreshToken) return res.status(401).json({ error: 'No refresh token' });
  try {
    const { sub } = jwt.verify(refreshToken, jwtRefreshSecret);
    res.json({ accessToken: signAccessToken(sub) });
  } catch {
    res.status(401).json({ error: 'Invalid refresh token' });
  }
});
```

#### Do's and Don'ts

**Do's**
- Keep access tokens short-lived (5ŌĆō15 min) to limit the damage window if one is stolen.
- Store refresh tokens in `httpOnly`, `Secure`, `SameSite=Strict` cookies ŌĆö not in `localStorage`.
- Validate the `issuer`, `audience`, and `expiresIn` claims during verification.

**Don'ts**
- Don't store sensitive data (passwords, payment info) in the JWT payload ŌĆö it is Base64-encoded, not encrypted.
- Never use the `none` algorithm ŌĆö it disables signature verification entirely.
- Avoid signing with a weak or hardcoded secret ŌĆö use a long random string from `crypto.randomBytes(64)`.

#### Common Follow-up Questions
- How do you revoke a JWT before it expires (e.g., on logout)?
- What is the difference between signing with a symmetric secret vs an asymmetric key pair (RS256)?
- When would you choose sessions over JWTs?

### **35. Database Connection Pooling**

> **Level:** Intermediate

#### Why This Is Asked
Creating a new database connection per request is one of the most common performance mistakes in Node.js backends ŌĆö this tests whether the candidate understands why pooling is essential.

#### Explanation
Establishing a database connection involves TCP handshakes, authentication, and memory allocation ŌĆö it can take 20ŌĆō100 ms. Creating one per request is slow and quickly exhausts the database's connection limit. A **connection pool** maintains a set of pre-established connections and lends them to requests, returning them when done. Pool size is a trade-off: too small causes queuing; too large overwhelms the DB. A good starting point is `(num_cores ├Ś 2) + spindle_count` ŌĆö typically 10ŌĆō20 per Node.js instance.

#### Code Example
```javascript
// PostgreSQL with pg-pool
const { Pool } = require('pg');

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 20,              // maximum connections in pool
  idleTimeoutMillis: 30_000,  // close idle connections after 30 s
  connectionTimeoutMillis: 2_000, // error if no connection available in 2 s
  ssl: process.env.NODE_ENV === 'production' ? { rejectUnauthorized: true } : false,
});

// The pool manages connection checkout/return automatically
async function getUser(id) {
  const { rows } = await pool.query('SELECT * FROM users WHERE id = $1', [id]);
  return rows[0] ?? null;
}

// Transaction ŌĆö must use a single client for the duration
async function transferFunds(fromId, toId, amount) {
  const client = await pool.connect(); // explicitly checkout one client
  try {
    await client.query('BEGIN');
    await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [amount, fromId]);
    await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [amount, toId]);
    await client.query('COMMIT');
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release(); // always return the client to the pool
  }
}

// Monitor pool health
pool.on('error', (err) => console.error('Idle pool client error:', err));
```

#### Do's and Don'ts

**Do's**
- Configure `connectionTimeoutMillis` ŌĆö without it, a saturated pool blocks callers indefinitely.
- Always `release()` a checked-out client in a `finally` block ŌĆö a leaked client starves the pool.
- Monitor pool metrics (`pool.totalCount`, `pool.idleCount`, `pool.waitingCount`) and alert on saturation.

**Don'ts**
- Don't create a new connection (or Pool) per request ŌĆö create the pool once at startup.
- Don't set `max` arbitrarily high ŌĆö each connection consumes memory on the DB server; check `max_connections` in PostgreSQL.
- Avoid using the pool directly for transactions ŌĆö check out a single client so all statements share the same connection.

#### Common Follow-up Questions
- How do you handle a pool that is consistently at max connections?
- What happens to queued requests when the DB goes down temporarily?
- How does connection pooling differ between PostgreSQL (`pg-pool`) and MongoDB (`mongoose`)?

### **36. Caching with Redis in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Caching is one of the highest-leverage performance tools available ŌĆö this tests whether the candidate can reason about cache patterns, invalidation, and the consistency trade-offs.

#### Explanation
The most common pattern is **cache-aside** (lazy loading): check the cache first; on a miss, fetch from the DB, write to cache, return the result. Redis is the standard external cache in Node.js because it supports TTL-based expiry, atomic operations, pub/sub, and data structures beyond key-value. Cache invalidation ŌĆö deciding when to expire or update cached data ŌĆö is the hard part: TTL-based expiry is simple but may serve stale data; event-based invalidation is precise but complex.

#### Code Example
```javascript
const { createClient } = require('redis');

const redis = createClient({ url: process.env.REDIS_URL });
await redis.connect();

redis.on('error', (err) => console.error('Redis error:', err));

// Cache-aside helper
async function cached(key, ttlSeconds, fetchFn) {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const data = await fetchFn();
  await redis.set(key, JSON.stringify(data), { EX: ttlSeconds });
  return data;
}

// Usage in a route
app.get('/products/:id', async (req, res) => {
  const product = await cached(
    `product:${req.params.id}`,
    300, // TTL: 5 minutes
    () => db.products.findById(req.params.id)
  );
  if (!product) return res.status(404).json({ error: 'Not found' });
  res.json(product);
});

// Cache invalidation on update
app.put('/products/:id', async (req, res) => {
  const updated = await db.products.update(req.params.id, req.body);
  await redis.del(`product:${req.params.id}`); // bust the cache
  res.json(updated);
});

// Avoid cache stampede: use SET NX (set if not exists) as a distributed lock
async function getWithLock(key, ttl, fetchFn) {
  const lockKey = `lock:${key}`;
  const locked = await redis.set(lockKey, '1', { NX: true, EX: 5 });
  if (!locked) {
    await new Promise((r) => setTimeout(r, 100));
    return cached(key, ttl, fetchFn);
  }
  const data = await fetchFn();
  await redis.set(key, JSON.stringify(data), { EX: ttl });
  await redis.del(lockKey);
  return data;
}
```

#### Do's and Don'ts

**Do's**
- Always set a TTL on cached values ŌĆö unbounded caches grow forever and eventually exhaust memory.
- Cache at the right granularity ŌĆö cache the final serialised response or individual entities, not raw DB rows.
- Handle Redis connection errors gracefully ŌĆö if Redis is unavailable, fall through to the DB rather than crashing.

**Don'ts**
- Don't cache user-specific data under a shared key ŌĆö always include the user ID in the cache key.
- Avoid caching mutable data without an invalidation strategy ŌĆö stale data is often worse than a cache miss.
- Don't cache errors ŌĆö a failed DB query should not be cached; the next request should retry.

#### Common Follow-up Questions
- What is a cache stampede and how do you prevent it?
- What is the difference between cache-aside, read-through, and write-through caching?
- How would you invalidate cache entries across multiple keys when a related record changes?

### **37. Rate Limiting in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Unprotected APIs are trivially abused ŌĆö rate limiting is a basic security and reliability control that interviewers expect production-minded candidates to know.

#### Explanation
Rate limiting caps the number of requests a client can make in a given time window. Common algorithms: **fixed window** (simple, but allows burst at window boundary), **sliding window** (smoother, no boundary burst), and **token bucket** (allows controlled bursting). `express-rate-limit` is the standard library for Express; it supports in-memory (single instance) and Redis-backed stores (multi-instance). Rate limit headers (`RateLimit-Remaining`, `Retry-After`) should be returned to help well-behaved clients back off.

#### Code Example
```javascript
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');
const { createClient } = require('redis');

const redisClient = createClient({ url: process.env.REDIS_URL });
await redisClient.connect();

// General API rate limit
const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15-minute window
  max: 100,                  // 100 requests per window per IP
  standardHeaders: 'draft-7', // return RateLimit headers
  legacyHeaders: false,
  store: new RedisStore({ sendCommand: (...args) => redisClient.sendCommand(args) }),
  message: { error: 'Too many requests ŌĆö please try again in 15 minutes' },
});

// Stricter limit for auth endpoints (prevent brute-force)
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 10,
  skipSuccessfulRequests: true, // only count failed attempts
  store: new RedisStore({ sendCommand: (...args) => redisClient.sendCommand(args) }),
});

app.use('/api/', apiLimiter);
app.use('/auth/login', authLimiter);
app.use('/auth/forgot-password', authLimiter);

// Rate limit by user ID (authenticated routes) instead of IP
const userLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 30,
  keyGenerator: (req) => req.user?.id ?? req.ip, // use user ID after auth
});
app.use('/api/user/', authenticate, userLimiter);
```

#### Do's and Don'ts

**Do's**
- Use a Redis-backed store in multi-instance deployments ŌĆö in-memory limits apply per process, not globally.
- Apply stricter limits to auth endpoints ŌĆö login, password reset, and OTP verification are primary brute-force targets.
- Return `Retry-After` headers so legitimate clients know when to retry without hammering the API.

**Don'ts**
- Don't rate limit only by IP in production ŌĆö shared IPs (corporate NAT, VPNs) would incorrectly block many users.
- Avoid using the default in-memory store in production ŌĆö limits reset on process restart and don't work across instances.
- Don't set limits so low that legitimate power users hit them ŌĆö monitor actual usage patterns first.

#### Common Follow-up Questions
- What is the difference between fixed window and sliding window rate limiting?
- How would you implement rate limiting per user rather than per IP?
- What is a token bucket algorithm and when is it preferable to a sliding window?

### **38. Worker Threads in Node.js**

> **Level:** Advanced

#### Why This Is Asked
Worker threads are the correct solution for CPU-bound work in Node.js ŌĆö this tests whether the candidate can articulate when to use them vs. `child_process`, and how data is passed between threads.

#### Explanation
`worker_threads` (available since Node.js v10.5, stable since v12) runs JavaScript in a separate OS thread with its own V8 instance and event loop, but within the same process. Unlike `child_process.fork`, workers share memory via `SharedArrayBuffer` and can transfer `ArrayBuffer`s with zero-copy using the `transferList`. Workers are best for CPU-bound tasks (image processing, cryptography, JSON parsing of large payloads) that would otherwise block the main thread's event loop.

#### Code Example
```javascript
// main.js ŌĆö spawns a worker for CPU-intensive work
const { Worker, isMainThread, parentPort, workerData } = require('worker_threads');
const path = require('path');

if (isMainThread) {
  function runWorker(data) {
    return new Promise((resolve, reject) => {
      const worker = new Worker(__filename, { workerData: data });
      worker.on('message', resolve);
      worker.on('error', reject);
      worker.on('exit', (code) => {
        if (code !== 0) reject(new Error(`Worker exited with code ${code}`));
      });
    });
  }

  // Main thread stays free for I/O while worker crunches numbers
  app.post('/process', async (req, res) => {
    const result = await runWorker({ payload: req.body.data });
    res.json({ result });
  });

} else {
  // Worker thread ŌĆö runs in a separate thread
  function heavyComputation(data) {
    let result = 0;
    for (let i = 0; i < 1e9; i++) result += i;
    return result;
  }

  const result = heavyComputation(workerData.payload);
  parentPort.postMessage(result);
}
```

```javascript
// Worker pool pattern ŌĆö reuse workers instead of spawning per request
const { StaticPool } = require('node-worker-threads-pool');

const pool = new StaticPool({
  size: require('os').cpus().length,
  task: './heavy-task.js',
});

app.post('/compute', async (req, res) => {
  const result = await pool.exec(req.body.data);
  res.json({ result });
});
```

#### Do's and Don'ts

**Do's**
- Use a worker pool (e.g., `node-worker-threads-pool`, `piscina`) instead of spawning a new worker per request ŌĆö thread creation is expensive.
- Transfer `ArrayBuffer`s using `transferList` for zero-copy passing of large binary data.
- Keep workers stateless ŌĆö send data in, get a result out. Stateful workers are hard to manage.

**Don'ts**
- Don't use `worker_threads` for I/O-bound work ŌĆö the event loop handles that efficiently already.
- Avoid sharing mutable state via `SharedArrayBuffer` without `Atomics` ŌĆö concurrent writes cause data races.
- Don't spawn more workers than CPU cores ŌĆö extra workers context-switch without adding throughput.

#### Common Follow-up Questions
- What is the difference between `worker_threads` and `child_process.fork`?
- How do you pass data between the main thread and a worker thread?
- When would you use `SharedArrayBuffer` with `Atomics` instead of message passing?

### **39. Structured Logging in Node.js**

> **Level:** Intermediate

#### Why This Is Asked
Production debugging relies entirely on logs ŌĆö this tests whether the candidate writes logs that are actually useful in a distributed system rather than `console.log` strings.

#### Explanation
**Structured logging** emits each log entry as a JSON object with consistent fields (level, timestamp, message, request ID, user ID, error details) rather than a plain string. This makes logs searchable and aggregatable in tools like Datadog, Grafana Loki, or ELK. **Pino** is the fastest Node.js logger (low overhead JSON serialisation); **Winston** is more configurable with multiple transports. Both support log levels (`trace`, `debug`, `info`, `warn`, `error`, `fatal`) that let you filter verbosity by environment.

#### Code Example
```javascript
// Pino ŌĆö fast, low-overhead structured logger
const pino = require('pino');

const logger = pino({
  level: process.env.LOG_LEVEL ?? 'info',
  // Pretty-print in development, JSON in production
  transport: process.env.NODE_ENV !== 'production'
    ? { target: 'pino-pretty', options: { colorize: true } }
    : undefined,
  // Redact sensitive fields from logs
  redact: { paths: ['req.headers.authorization', 'body.password'], censor: '[REDACTED]' },
});

// Request logger middleware ŌĆö attach correlation ID
const { randomUUID } = require('crypto');
app.use((req, res, next) => {
  req.id = req.headers['x-request-id'] ?? randomUUID();
  req.log = logger.child({ requestId: req.id, method: req.method, url: req.url });
  req.log.info('Request received');

  const start = Date.now();
  res.on('finish', () => {
    req.log.info({ statusCode: res.statusCode, durationMs: Date.now() - start }, 'Request completed');
  });
  next();
});

// Use contextual child loggers ŌĆö carry request context automatically
app.get('/users/:id', async (req, res) => {
  req.log.debug({ userId: req.params.id }, 'Fetching user');
  try {
    const user = await db.findById(req.params.id);
    req.log.info({ userId: user.id }, 'User found');
    res.json(user);
  } catch (err) {
    req.log.error({ err }, 'Failed to fetch user');
    res.status(500).json({ error: 'Internal server error' });
  }
});
```

#### Do's and Don'ts

**Do's**
- Attach a unique `requestId` (correlation ID) to every log entry so you can trace a full request across services.
- Log at the right level ŌĆö `debug` for detailed internals, `info` for significant events, `warn` for recoverable issues, `error` for failures.
- Use `redact` to automatically scrub sensitive fields (tokens, passwords, PII) before they hit the log transport.

**Don'ts**
- Don't use `console.log` in production ŌĆö it produces unstructured strings that cannot be queried.
- Avoid logging inside tight loops or on every DB query in production ŌĆö high-volume logging becomes a measurable I/O bottleneck.
- Don't log sensitive data (passwords, credit card numbers, session tokens) even accidentally ŌĆö use a redaction list.

### **40. Testing Node.js Applications**

> **Level:** Intermediate

#### Why This Is Asked
Testing discipline separates candidates who write throwaway code from those who build maintainable systems ŌĆö this tests knowledge of testing levels, tooling, and what to actually test.

#### Explanation
Node.js testing splits into three levels: **unit tests** (test a single function in isolation ŌĆö fast, no I/O), **integration tests** (test a module interacting with real or realistic dependencies like a DB), and **end-to-end tests** (test the full HTTP surface). **Jest** is the dominant test runner for both unit and integration; **Supertest** wraps Express to fire real HTTP requests in tests without a running server. Mocking should be used sparingly ŌĆö over-mocked tests pass when the real integration is broken.

#### Code Example
```javascript
// userService.js
async function getUserById(db, id) {
  if (!id) throw new Error('ID is required');
  return db.query('SELECT * FROM users WHERE id = $1', [id]);
}
module.exports = { getUserById };

// userService.test.js ŌĆö unit test (no real DB)
const { getUserById } = require('./userService');

describe('getUserById', () => {
  it('throws when ID is missing', async () => {
    await expect(getUserById({}, null)).rejects.toThrow('ID is required');
  });

  it('queries the database with the correct ID', async () => {
    const mockDb = { query: jest.fn().mockResolvedValue({ rows: [{ id: 1, name: 'Alice' }] }) };
    const result = await getUserById(mockDb, 1);
    expect(mockDb.query).toHaveBeenCalledWith(expect.stringContaining('WHERE id'), [1]);
    expect(result.rows[0].name).toBe('Alice');
  });
});

// app.test.js ŌĆö integration test with Supertest (real Express, real DB)
const request = require('supertest');
const app = require('./app');
const db = require('./db');

beforeAll(() => db.migrate.latest());
afterAll(() => db.destroy());
afterEach(() => db('users').truncate());

describe('GET /users/:id', () => {
  it('returns 200 with user data', async () => {
    const [id] = await db('users').insert({ name: 'Alice' }).returning('id');

    const res = await request(app).get(`/users/${id}`).expect(200);
    expect(res.body.name).toBe('Alice');
  });

  it('returns 404 for unknown user', async () => {
    await request(app).get('/users/99999').expect(404);
  });
});
```

#### Do's and Don'ts

**Do's**
- Write integration tests against a real test database ŌĆö mocked DB tests miss SQL errors, constraint violations, and query bugs.
- Use `beforeAll`/`afterAll` for expensive setup (migrations) and `afterEach` to reset data between tests.
- Test the HTTP contract (status codes, response shape) not just the internal functions ŌĆö the route is what users actually call.

**Don'ts**
- Don't mock everything ŌĆö over-mocking produces tests that pass while the actual integration is broken.
- Avoid testing implementation details (which internal functions were called) ŌĆö test behaviour and output instead.
- Don't run integration tests against a shared development or production database ŌĆö use an isolated test DB, ideally spun up in CI via Docker.

- What is the difference between a mock, a stub, and a spy in Jest?
- How would you test a route that requires authentication?
- When would you use snapshot testing and when would it be a bad fit?

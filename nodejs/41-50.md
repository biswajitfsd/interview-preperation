### **41. Node.js CPU & Heap Profiling — Deep Dive**

> **Level:** Advanced

#### Why This Is Asked
HighLevel processes 4 billion API hits per day across 250+ Node.js microservices. A 10 ms regression in a hot-path function, multiplied across millions of calls, accumulates into real user-visible latency. Lead engineers must diagnose bottlenecks systematically using profiling tools — not guessing or micro-optimising blindly.

#### Explanation

**CPU Profiling** samples the V8 call stack at a fixed interval (typically every 100 µs). The result is a call graph showing which functions consumed the most CPU time. A flame graph visualises this: the x-axis is time (wider = more CPU), the y-axis is the call stack (deeper = more nesting). Hot paths appear as wide, tall plateaus.

**Heap Profiling** tracks object allocations over time. Unlike a heap snapshot (a point-in-time view of live objects), a heap sampling profile records *where* objects are being allocated by sampling every N bytes of allocation. Two heap snapshots taken minutes apart let you compare retained objects — anything that grew between the two snapshots is a candidate for a memory issue.

**Event Loop Lag** is a third critical signal. The event loop is single-threaded; if a synchronous operation blocks it for even 50 ms, all concurrent requests queue behind it. You measure lag by scheduling a `setImmediate` callback and measuring how long it takes to fire.

The tooling hierarchy:
- **Built-in**: `--prof` flag generates a V8 isolate log; `node --prof-process` converts it to text. `--inspect` enables the Chrome DevTools Protocol for live profiling.
- **CLI tools**: `0x` generates an interactive flame graph HTML. `clinic.js` provides three specialised tools.
- **Programmatic**: the `inspector` and `v8` modules allow triggering profiles and snapshots from within the process (useful for production heap dumps via an internal endpoint).

#### Code Example

```javascript
// ── Tool 1: Built-in V8 CPU profiler ──────────────────────────────────────
// node --prof app.js            → produces isolate-0x...log
// node --prof-process isolate-0x...log > cpu.txt   → human-readable

// ── Tool 2: 0x (interactive flame graph) ──────────────────────────────────
// npx 0x -- node app.js
// Generates .html — open in browser; click frames to zoom in

// ── Tool 3: Clinic.js ─────────────────────────────────────────────────────
// npx clinic flame    -- node app.js   # CPU flame graph (same as 0x)
// npx clinic bubbleprof -- node app.js # async operation map (I/O patterns)
// npx clinic doctor   -- node app.js   # auto-detect: slow event loop? I/O bottleneck?

// ── Programmatic heap snapshot (expose via authenticated internal endpoint) ─
const v8 = require('v8');
const path = require('path');

let snapshotSeq = 0;

// Only accessible from inside the cluster (use NetworkPolicy or IP allowlist)
app.post('/_internal/heap-snapshot', requireInternalToken, (req, res) => {
  const file = path.join('/tmp', `heap-${Date.now()}-${++snapshotSeq}.heapsnapshot`);
  v8.writeHeapSnapshot(file);
  res.json({ file });
  // Download the file and load it in Chrome DevTools → Memory tab → Load snapshot
});

// ── Programmatic heap sampling profile ────────────────────────────────────
const inspector = require('inspector');
const fs = require('fs');

app.post('/_internal/heap-profile/start', requireInternalToken, (req, res) => {
  const session = new inspector.Session();
  session.connect();
  session.post('HeapProfiler.startSampling', { samplingInterval: 32768 }, (err) => {
    if (err) return res.status(500).json({ error: err.message });
    res.locals.session = session;
    res.json({ started: true });
  });
});

app.post('/_internal/heap-profile/stop', requireInternalToken, (req, res) => {
  // Assumes session was stored (simplified — in production use a singleton)
  const session = req.app.locals.activeSession;
  session.post('HeapProfiler.stopSampling', (err, { profile }) => {
    const file = `/tmp/heap-${Date.now()}.heapprofile`;
    fs.writeFileSync(file, JSON.stringify(profile));
    session.disconnect();
    res.json({ file });
    // Open .heapprofile in Chrome DevTools → Memory → Load Allocation Profile
  });
});

// ── Event Loop Lag monitoring ──────────────────────────────────────────────
const { performance } = require('perf_hooks');
const promClient = require('prom-client');

const eventLoopLag = new promClient.Gauge({
  name: 'node_event_loop_lag_ms',
  help: 'Node.js event loop lag in milliseconds',
});

const eventLoopUtilization = new promClient.Gauge({
  name: 'node_event_loop_utilization',
  help: 'Node.js event loop utilization (0-1)',
});

let previousELU = performance.eventLoopUtilization();

setInterval(() => {
  // Measure lag via setImmediate round-trip
  const start = process.hrtime.bigint();
  setImmediate(() => {
    const lag = Number(process.hrtime.bigint() - start) / 1e6;
    eventLoopLag.set(lag);
    if (lag > 200) {
      logger.warn({ lagMs: lag }, 'Event loop lag spike — possible blocking operation');
    }
  });

  // Measure utilization: fraction of time the event loop was active
  const current = performance.eventLoopUtilization(previousELU);
  previousELU = performance.eventLoopUtilization();
  eventLoopUtilization.set(current.utilization);
  // > 0.85 means the loop is busy 85% of the time — approaching saturation
}, 5000);
```

#### Reading a Flame Graph — Key Patterns

| Pattern | What it means |
|---|---|
| Wide plateau at the top | Hot path — the function consuming the most CPU |
| Wide frames in the middle | Intermediary functions called many times |
| Very deep stack, narrow | Deep recursion — not necessarily slow unless wide |
| V8 internals (yellow in 0x) | GC pressure, JIT compilation overhead |
| `JSON.parse` / `JSON.stringify` wide | Serialisation is a bottleneck — consider binary formats |

#### Do's and Don'ts

**Do's**
- Profile under realistic production-like load — use `k6` or `Artillery` to generate traffic while the profiler runs. Cold-server profiles are meaningless.
- Take two heap snapshots 5 minutes apart; compare them in Chrome DevTools Memory tab — the "Delta" column reveals retained growth.
- Use `--inspect` with a dedicated diagnostic port (`--inspect=9230`) and restrict access via firewall, not the default `:9229` shared with debuggers.

**Don'ts**
- Don't optimise code based on a profiler run with only one or two requests — the V8 JIT hasn't warmed up yet. Run at least 1,000 requests before reading the flame graph.
- Avoid running `v8.writeHeapSnapshot()` in production under live traffic without testing — it pauses GC and can spike latency for 100–500 ms on a large heap.
- Don't confuse high heap *usage* with a leak — a Node.js process holds freed memory in a pool before returning it to the OS. Rising `heapUsed` after GC cycles is the real signal.

#### Common Follow-up Questions
- You see a wide `JSON.stringify` frame covering 30% of CPU time in a flame graph. What would you do?
- How do you profile a single slow endpoint in production without profiling every request?
- What does a "flat" flame graph (all frames roughly the same width across the bottom) indicate about the bottleneck?

### **42. Memory Leak Detection & Prevention**

> **Level:** Advanced

#### Why This Is Asked
Memory leaks in a long-running Node.js service cause a slow upward trend in RSS (Resident Set Size). Without proactive detection, pods are OOM-killed by Kubernetes, causing cascading restarts across the fleet. At HighLevel's scale — 250+ microservices — undetected leaks compound into platform-wide instability.

#### Explanation

A memory leak in Node.js means objects that should be garbage-collected remain reachable (directly or through a reference chain), so the V8 GC cannot free them. The four most common causes are:

1. **EventEmitter listeners added but never removed** — each `.on()` call attaches a closure that holds a reference to the enclosing scope. Inside a request handler that runs millions of times, this is catastrophic.
2. **Unbounded in-process caches** — `const cache = new Map()` at module scope grows forever unless you evict old entries.
3. **Timers not cleared** — `setInterval` keeps running even if the object that created it is no longer referenced.
4. **Promises that never settle** — a pending Promise and its closure stay in memory until the Promise resolves or rejects. A hanging HTTP call that never times out is a common source.

Detection workflow: confirm the leak with memory metrics, isolate it with heap snapshot comparison, identify the retaining path in Chrome DevTools.

#### Code Example

```javascript
// ── Common leak patterns and their fixes ──────────────────────────────────

// LEAK 1: EventEmitter listener added on every request
class RequestHandler {
  handle(req) {
    // Every call adds a new listener — they accumulate forever
    globalEmitter.on('flush', () => this.flush(req));
  }
}

// FIX: Use .once() or remove the listener when done
class RequestHandler {
  handle(req) {
    const onFlush = () => this.flush(req);
    globalEmitter.once('flush', onFlush);
    // Or: req.on('close', () => globalEmitter.off('flush', onFlush));
  }
}

// LEAK 2: Unbounded module-level cache
const resultCache = {}; // grows forever

// FIX: Use lru-cache with a size limit
const LRU = require('lru-cache');
const resultCache = new LRU({ max: 500, ttl: 1000 * 60 * 5 }); // 500 entries, 5 min TTL

// LEAK 3: Timer not cleared
function startPoll(resource) {
  const interval = setInterval(() => resource.poll(), 1000);
  // If resource is GC'd, interval keeps running — resource is never collected
}

// FIX: Store and clear the interval; call .unref() to not block process exit
function startPoll(resource) {
  const interval = setInterval(() => resource.poll(), 1000);
  interval.unref(); // won't prevent process exit
  resource.on('close', () => clearInterval(interval));
  return interval;
}

// ── RSS monitoring with alerts ─────────────────────────────────────────────
const promClient = require('prom-client');

const heapUsedGauge = new promClient.Gauge({
  name: 'node_heap_used_bytes',
  help: 'V8 heap used bytes',
});
const rssGauge = new promClient.Gauge({
  name: 'node_rss_bytes',
  help: 'Process RSS bytes',
});

const baselineRss = process.memoryUsage().rss;

setInterval(() => {
  const { rss, heapUsed, heapTotal, external } = process.memoryUsage();
  heapUsedGauge.set(heapUsed);
  rssGauge.set(rss);

  const rssGrowthMb = (rss - baselineRss) / 1024 / 1024;
  if (rssGrowthMb > 400) {
    logger.error({ rssGrowthMb, heapUsed, heapTotal }, 'Possible memory leak — RSS grown 400 MB from baseline');
  }
}, 15_000);

// ── On-demand heap snapshot endpoint (internal only) ───────────────────────
const v8 = require('v8');

app.post('/_internal/gc', requireInternalToken, (req, res) => {
  // Force a full GC before taking a snapshot to reduce noise
  if (global.gc) {
    global.gc(); // requires --expose-gc flag
    res.json({ gc: 'triggered' });
  } else {
    res.json({ gc: 'not available — start with --expose-gc' });
  }
});

app.post('/_internal/heap-snapshot', requireInternalToken, (req, res) => {
  const file = v8.writeHeapSnapshot('/tmp');
  res.json({ file });
  // Workflow: gc → snapshot A → wait 5 min under load → gc → snapshot B
  // Load both in Chrome DevTools Memory → Compare → sort Delta descending
  // The objects with highest positive Delta are leak candidates
  // Right-click any object → Retaining tree → shows what holds the reference
});
```

#### Leak Investigation Workflow

```
1. CONFIRM  — Graph process.memoryUsage().heapUsed over 30 min.
              Steady upward trend after GC cycles = leak confirmed.

2. ISOLATE  — Trigger GC (--expose-gc), take Snapshot A.
              Apply 5 min of load. Trigger GC again. Take Snapshot B.

3. COMPARE  — Chrome DevTools Memory → Load B → "Compared to Snapshot A"
              Sort by "Delta" descending → large positive numbers = accumulated objects.

4. TRACE    — Right-click any leaked object type → Retaining Path.
              Follow the chain back to the GC root — that's your leak source.

5. FIX      — Remove the retaining reference:
                - EventEmitter: call .off() or .removeAllListeners()
                - Cache: add eviction policy
                - Timer: call clearInterval / clearTimeout
                - Promise: add timeout + abort signal

6. VERIFY   — Deploy fix. Graph RSS over 24 h — should plateau, not grow.
```

#### Do's and Don'ts

**Do's**
- Use `lru-cache` for all in-process caches — it automatically evicts the least-recently-used entry and supports TTL.
- Monitor both `heapUsed` and `rss` separately — `rss` includes memory V8 has freed but not yet returned to the OS; `heapUsed` is the truer leak signal.
- Enable Node.js's built-in leak warning: `EventEmitter.defaultMaxListeners = 15` — if a single emitter exceeds 15 listeners, it prints a warning.

**Don'ts**
- Don't suppress the `MaxListenersExceededWarning` with `emitter.setMaxListeners(Infinity)` — it is a canary in the coal mine. Investigate instead.
- Never use `global` as a cache — global objects are GC roots and are never collected.
- Don't restart the pod as a "fix" for a memory leak — it just delays the OOM kill. Find and fix the root cause.

#### Common Follow-up Questions
- Your Kubernetes pod is OOM-killed every 6 hours. The heap snapshot shows thousands of `IncomingMessage` objects. What does that tell you?
- How would you detect a leak in a service that uses `AsyncLocalStorage` for request context?
- What is the difference between external memory (shown in `process.memoryUsage().external`) and heap memory?

### **43. Circuit Breaker & Cascading Failure Prevention**

> **Level:** Advanced

#### Why This Is Asked
HighLevel runs 250+ microservices. One slow or unavailable dependency — a DB, a third-party API, a downstream service — can cause all callers to queue requests, exhaust their connection pools, and crash. This cascade can take down dozens of unrelated services within minutes. Circuit breakers are the standard industry defence.

#### Explanation

A **circuit breaker** wraps calls to an external dependency and monitors the failure rate. It has three states:

- **CLOSED** — normal operation; requests pass through
- **OPEN** — dependency is failing; requests fail immediately without calling the dependency (fast failure)
- **HALF-OPEN** — after a reset timeout, one probe request is allowed; if it succeeds the circuit closes; if it fails it opens again

The key parameters are: failure rate threshold (open when X% of calls fail in a window), minimum volume threshold (don't open on the first two failures), and reset timeout (how long to stay open before probing).

**Bulkhead** is a complementary pattern: limit the number of concurrent calls to each downstream service so one slow service can't consume all threads/connections. Named after watertight compartments in ships — if one floods, others stay dry.

**Retry with exponential backoff and jitter** prevents retry storms: if 1,000 clients all retry at the same time after a failure, they create a thundering herd that prevents recovery.

#### Code Example

```javascript
const CircuitBreaker = require('opossum');

// ── Wrapping a downstream service call ────────────────────────────────────
async function callPaymentService(payload) {
  const res = await fetch('http://payment-service/v1/charge', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(payload),
    signal: AbortSignal.timeout(2000), // hard 2 s timeout
  });
  if (!res.ok) throw new Error(`PaymentService HTTP ${res.status}`);
  return res.json();
}

const paymentBreaker = new CircuitBreaker(callPaymentService, {
  timeout: 3000,                   // treat calls > 3 s as failures
  errorThresholdPercentage: 50,    // open when 50% of calls fail
  volumeThreshold: 10,             // need at least 10 calls to evaluate
  resetTimeout: 30_000,            // wait 30 s before probing (HALF-OPEN)
  name: 'payment-service',
});

// Fallback: what to return when the circuit is OPEN
paymentBreaker.fallback((payload) => ({
  status: 'deferred',
  message: 'Payment will be processed once service recovers',
  deferredId: crypto.randomUUID(),
}));

// Observability: emit circuit state as metrics
const promClient = require('prom-client');
const circuitState = new promClient.Gauge({
  name: 'circuit_breaker_state',
  help: 'Circuit breaker state: 0=closed, 1=open, 2=half-open',
  labelNames: ['name'],
});

paymentBreaker.on('open',     () => {
  circuitState.labels('payment-service').set(1);
  logger.warn('Circuit OPEN: payment-service failing');
});
paymentBreaker.on('halfOpen', () => {
  circuitState.labels('payment-service').set(2);
  logger.info('Circuit HALF-OPEN: probing payment-service');
});
paymentBreaker.on('close',    () => {
  circuitState.labels('payment-service').set(0);
  logger.info('Circuit CLOSED: payment-service recovered');
});
paymentBreaker.on('fallback', (result) =>
  logger.warn({ result }, 'Circuit fallback triggered'));

// Usage
app.post('/checkout', async (req, res) => {
  try {
    const result = await paymentBreaker.fire(req.body.payment);
    res.json(result);
  } catch (err) {
    // Breaker itself can throw if fallback also fails
    res.status(503).json({ error: 'Service temporarily unavailable' });
  }
});

// ── Bulkhead: limit concurrent calls per dependency ────────────────────────
const pLimit = require('p-limit');

// Max 15 concurrent calls to the enrichment service at any one time
const enrichmentLimiter = pLimit(15);

async function enrichContacts(contactIds) {
  return Promise.all(
    contactIds.map((id) =>
      enrichmentLimiter(() => enrichmentService.enrich(id))
    )
  );
}

// ── Retry with exponential backoff + jitter ────────────────────────────────
async function withRetry(fn, { maxAttempts = 3, baseDelayMs = 500 } = {}) {
  let lastError;
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err) {
      lastError = err;
      // Don't retry client errors (4xx) — they won't resolve
      if (err.status >= 400 && err.status < 500) throw err;
      if (attempt === maxAttempts) break;

      // Exponential backoff with full jitter — prevents thundering herd
      const cap = Math.min(baseDelayMs * 2 ** attempt, 10_000);
      const delay = Math.random() * cap;
      logger.warn({ attempt, delayMs: delay, err: err.message }, 'Retrying after error');
      await new Promise((r) => setTimeout(r, delay));
    }
  }
  throw lastError;
}
```

#### Do's and Don'ts

**Do's**
- Pair every circuit breaker with a meaningful fallback — an open circuit with no fallback just converts slow failures into fast errors with no user value.
- Export circuit state as a Prometheus metric and create a Grafana dashboard showing which circuits are open across all services. An open circuit is a leading indicator of an incident.
- Set `timeout` on the circuit breaker lower than your SLO latency target — fail fast, don't hold open connections.

**Don'ts**
- Don't retry non-idempotent operations (payment charges, order creation) without a deduplication key — a successful first attempt followed by a retry creates a duplicate.
- Never retry 4xx responses — a `400 Bad Request` or `404 Not Found` will not succeed on retry; it indicates a client-side problem.
- Don't set `errorThresholdPercentage` too low (e.g., 5%) — a single transient error shouldn't open the circuit. Use `volumeThreshold` to require a minimum sample size.

#### Common Follow-up Questions
- What is the difference between a circuit breaker and a retry? Should they be combined?
- If the fallback function also throws, what happens? How do you handle a failed fallback?
- How would you test circuit breaker behaviour in a CI environment without taking a real service down?

### **44. Microservices Communication Patterns**

> **Level:** Advanced

#### Why This Is Asked
HighLevel runs 250+ microservices handling 4 billion API hits per day and 2.5 billion message events. The pattern chosen for inter-service communication directly determines latency, reliability, coupling, and operational complexity. A Lead Engineer must choose the right pattern for each interaction — not default to REST for everything.

#### Explanation

**Synchronous patterns** (REST, gRPC) require the caller to wait for a response. They are appropriate when the caller needs a result immediately and can tolerate blocking. REST is simple and debuggable; gRPC uses Protocol Buffers for binary serialisation and supports streaming — it is 3–10× more efficient than REST for high-frequency internal calls.

**Asynchronous patterns** (Pub/Sub, Kafka, RabbitMQ) decouple producers from consumers. The producer emits an event and continues without waiting. This is appropriate for workflows where the outcome is not needed immediately (send an email, trigger an automation, sync data to ClickHouse). At-least-once delivery means consumers must handle duplicate messages idempotently.

**The Saga Pattern** coordinates multi-step workflows that span multiple services without distributed transactions. Each step publishes an event; if a step fails, compensating events roll back previous steps. There are two flavours: *choreography* (services react to events directly — simpler but harder to trace) and *orchestration* (a central coordinator issues commands — easier to trace but a single point of failure).

#### Code Example

```javascript
// ── REST (synchronous — for simple CRUD needing immediate response) ────────
const res = await fetch('http://contact-service/v1/contacts/abc', {
  headers: { Authorization: `Bearer ${internalToken}` },
  signal: AbortSignal.timeout(3000),
});
if (!res.ok) throw new Error(`Contact service: ${res.status}`);
const contact = await res.json();

// ── gRPC (synchronous — for high-frequency internal calls) ────────────────
// contact.proto:
// service ContactService { rpc GetContact(ContactRequest) returns (ContactResponse); }
const grpc = require('@grpc/grpc-js');
const protoLoader = require('@grpc/proto-loader');

const packageDef = protoLoader.loadSync('contact.proto');
const ContactService = grpc.loadPackageDefinition(packageDef).ContactService;

const client = new ContactService(
  'contact-service:50051',
  grpc.credentials.createInsecure(),
);

const contact = await new Promise((resolve, reject) => {
  client.GetContact({ id: 'abc' }, (err, response) => {
    if (err) reject(err); else resolve(response);
  });
});

// ── GCP Pub/Sub (async — for event-driven workflows) ───────────────────────
const { PubSub } = require('@google-cloud/pubsub');
const pubsub = new PubSub({ projectId: process.env.GCP_PROJECT });

// Publisher: emit an event after a contact is created
async function publishContactCreated(contact) {
  const event = {
    type: 'contact.created',
    version: 1,
    correlationId: crypto.randomUUID(),
    tenantId: contact.tenantId,
    payload: contact,
    timestamp: new Date().toISOString(),
  };
  const messageId = await pubsub
    .topic('crm-events')
    .publishMessage({ data: Buffer.from(JSON.stringify(event)) });
  logger.info({ messageId, contactId: contact.id }, 'Published contact.created');
  return messageId;
}

// Subscriber: react to the event (e.g., automation-worker)
const subscription = pubsub.subscription('automation-worker-crm-events', {
  flowControl: { maxMessages: 100 }, // backpressure: process max 100 at a time
});

subscription.on('message', async (message) => {
  const event = JSON.parse(message.data.toString());
  logger.info({ messageId: message.id, type: event.type }, 'Processing event');

  try {
    await handleEvent(event);
    message.ack(); // only ack AFTER successful processing
  } catch (err) {
    logger.error({ err, messageId: message.id }, 'Event processing failed');
    message.nack(); // nack → Pub/Sub redelivers (with backoff)
  }
});

subscription.on('error', (err) =>
  logger.error({ err }, 'Pub/Sub subscription error'));

// ── Choreography Saga (distributed workflow) ──────────────────────────────
// Order Service emits 'order.created'
// Payment Service listens, charges card, emits 'payment.charged' or 'payment.failed'
// Inventory Service listens to 'payment.charged', reserves stock
// If inventory fails, emits 'inventory.failed'
// Payment Service listens to 'inventory.failed', issues refund
// Each service only knows about events — no direct coupling

async function handleEvent(event) {
  switch (event.type) {
    case 'order.created':
      await chargeCard(event.payload);
      await publishPaymentCharged(event.payload);
      break;
    case 'inventory.failed':
      await issueRefund(event.payload.orderId);
      await publishPaymentRefunded(event.payload);
      break;
    default:
      logger.debug({ type: event.type }, 'Unhandled event type — ignoring');
  }
}
```

#### Pattern Selection Guide

| Scenario | Pattern | Why |
|---|---|---|
| Fetch user profile in a request handler | REST / gRPC | Caller needs result immediately |
| High-frequency internal microservice calls | gRPC | Binary protocol, lower overhead |
| Send welcome email after signup | Pub/Sub (async) | Not needed in the request path |
| Sync lead data to ClickHouse | Pub/Sub (async) | Decoupled, buffered, retryable |
| Multi-step checkout across 3 services | Saga (choreography) | No distributed transactions |
| Real-time dashboard data | gRPC streaming | Server-push without polling |

#### Do's and Don'ts

**Do's**
- Always include a `correlationId` and `tenantId` in every event so logs and traces can be correlated across services.
- Version your event schemas (e.g., `version: 1`) — consumers need to handle both old and new formats during rolling deployments.
- Use `flowControl: { maxMessages: N }` on Pub/Sub subscriptions to implement backpressure — without it, a slow consumer accumulates unprocessed messages in memory.

**Don'ts**
- Don't access another service's database directly — each service owns its data. Communicate via API or events.
- Avoid synchronous REST calls in the critical path when the result isn't needed immediately — use async events instead and respond to the client with a `202 Accepted`.
- Don't assume Pub/Sub delivers exactly once — messages can be redelivered. Design consumers to be idempotent.

#### Common Follow-up Questions
- When would you choose choreography over orchestration for a saga, and what are the trade-offs?

### **45. Distributed Tracing with OpenTelemetry**

> **Level:** Advanced

#### Why This Is Asked
A single API request at HighLevel can touch 8+ microservices. Without distributed tracing, a 3-second latency spike is invisible — you can't tell which service added the latency or why. OpenTelemetry (OTel) is the vendor-neutral CNCF standard that HighLevel's stack is likely built on.

#### Explanation

**Key concepts:**
- **Trace** — the complete journey of one request, from the edge to every downstream service it touched
- **Span** — a single unit of work within a trace (one DB query, one HTTP call, one function call). Spans form a tree.
- **Context propagation** — trace and span IDs travel between services via HTTP headers (`traceparent` W3C standard) or message attributes (for Pub/Sub). Without propagation, traces break at service boundaries.
- **Sampling** — recording every span is too expensive (4 billion hits/day = enormous trace storage). **Head sampling** decides at the first service entry (simple but can drop interesting traces). **Tail sampling** decides after the full trace is assembled — it can always capture traces with errors or high latency, discarding routine healthy traces.

OTel provides auto-instrumentation for HTTP, Express, MongoDB, Redis, and gRPC — zero code required for 80% of spans. Manual spans cover business-critical operations that auto-instrumentation won't capture.

#### Code Example

```javascript
// ── OTel SDK setup (MUST be the first require in your entry point) ─────────
// tracing.js — required before app.js
const { NodeSDK } = require('@opentelemetry/sdk-node');
const { OTLPTraceExporter } = require('@opentelemetry/exporter-trace-otlp-http');
const { getNodeAutoInstrumentations } = require('@opentelemetry/auto-instrumentations-node');
const { Resource } = require('@opentelemetry/resources');
const { SemanticResourceAttributes } = require('@opentelemetry/semantic-conventions');

const sdk = new NodeSDK({
  resource: new Resource({
    [SemanticResourceAttributes.SERVICE_NAME]: process.env.SERVICE_NAME || 'crm-service',
    [SemanticResourceAttributes.DEPLOYMENT_ENVIRONMENT]: process.env.NODE_ENV,
  }),
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT, // e.g. http://otel-collector:4318/v1/traces
  }),
  instrumentations: [
    getNodeAutoInstrumentations({
      '@opentelemetry/instrumentation-fs': { enabled: false }, // too noisy
    }),
  ],
});

sdk.start();
process.on('SIGTERM', () => sdk.shutdown());

// ── Manual spans for critical business operations ──────────────────────────
// app.js
const { trace, SpanStatusCode, SpanKind } = require('@opentelemetry/api');
const tracer = trace.getTracer('crm-service', '1.0.0');

async function processCustomObject(tenantId, objectType, data) {
  // Start a manual span for this business operation
  return tracer.startActiveSpan('custom_object.process', {
    kind: SpanKind.INTERNAL,
    attributes: {
      'tenant.id': tenantId,
      'object.type': objectType,
      'object.field_count': Object.keys(data).length,
    },
  }, async (span) => {
    try {
      // Validate against schema — auto-instrumentation will trace the DB call inside
      const schema = await schemaRegistry.getSchema(tenantId, objectType);
      validateData(schema, data);

      const record = await db.customObjects.insertOne({ tenantId, objectType, data });

      span.setAttribute('object.id', record.insertedId.toString());
      span.setStatus({ code: SpanStatusCode.OK });
      return record;
    } catch (err) {
      span.recordException(err); // attaches error stack to the span
      span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
      throw err;
    } finally {
      span.end(); // always end the span
    }
  });
}

// ── Context propagation through GCP Pub/Sub ───────────────────────────────
const { propagation, context } = require('@opentelemetry/api');

// Publisher: inject current trace context into message attributes
async function publishWithContext(topicName, data) {
  const carrier = {};
  propagation.inject(context.active(), carrier);
  // carrier now contains: { traceparent: '00-abc123...', tracestate: '...' }

  await pubsub.topic(topicName).publishMessage({
    data: Buffer.from(JSON.stringify(data)),
    attributes: carrier, // Pub/Sub message attributes carry trace context
  });
}

// Subscriber: extract trace context and continue the trace
subscription.on('message', async (message) => {
  const remoteCtx = propagation.extract(context.active(), message.attributes);

  await context.with(remoteCtx, async () => {
    return tracer.startActiveSpan('pubsub.message.process', async (span) => {
      try {
        span.setAttribute('messaging.system', 'pubsub');
        span.setAttribute('messaging.destination', subscription.name);

        const event = JSON.parse(message.data.toString());
        await handleEvent(event);
        message.ack();
        span.setStatus({ code: SpanStatusCode.OK });
      } catch (err) {
        span.recordException(err);
        span.setStatus({ code: SpanStatusCode.ERROR });
        message.nack();
      } finally {
        span.end();
      }
    });
  });
});
```

#### Sampling Strategy for HighLevel Scale

```
Head sampling (at entry service):
  - Sample 1% of all requests  ← too aggressive: drops most interesting traces

Tail sampling (at OTel Collector after full trace assembled):
  - ALWAYS sample: error rate > 0%
  - ALWAYS sample: latency > 2× p50 (slow outliers)
  - ALWAYS sample: new deployment window (first 30 min after deploy)
  - Sample 0.1% of healthy, fast traces  ← reduces storage by 99% while keeping all interesting traces
```

#### Do's and Don'ts

**Do's**
- Auto-instrument first — `getNodeAutoInstrumentations()` instruments HTTP, Express, MongoDB, Redis, and gRPC automatically. This covers 80% of spans with zero code changes.
- Add manual spans only for critical business operations (charge payment, trigger automation, validate schema) that auto-instrumentation doesn't cover.
- Use tail sampling in the OTel Collector — always capture error traces and high-latency traces, discard routine healthy traces.

**Don'ts**
- Don't add spans inside tight loops (e.g., span per array element in a batch) — this creates thousands of spans per trace and blows up storage costs.
- Avoid head sampling if you need to always capture error traces — errors are rare but important; head sampling can discard them.
- Don't forget to call `span.end()` — open spans are held in memory and sent as incomplete traces.

- How do you propagate trace context through a job queue (like BullMQ) that doesn't use HTTP headers?
- What is the difference between a span event and a span attribute? When would you use each?

### **46. RED/USE Metrics & SLO Engineering**

> **Level:** Advanced

#### Why This Is Asked
HighLevel's SRE and on-call engineers use structured metrics frameworks to diagnose incidents quickly. A Lead Engineer is expected to design services that emit the right signals, set meaningful SLOs, and reason about error budgets — not just "make the service faster."

#### Explanation

**RED** (Rate, Errors, Duration) is applied to every *service endpoint*. It answers: "Is this service working, and how fast?" RED metrics are the first thing you look at during an incident.

**USE** (Utilization, Saturation, Errors) is applied to every *resource* (CPU, memory, DB pool, event loop). It answers: "Is this resource a bottleneck?" If RED shows a service is slow, USE tells you which resource is causing it.

**SLI/SLO/Error Budget:**
- **SLI** (Service Level Indicator) — the metric being measured (e.g., "% of requests with latency < 500 ms")
- **SLO** (Service Level Objective) — the target (e.g., "99.9% of requests meet the SLI")
- **Error Budget** — `1 - SLO` = the amount of failure allowed (0.1% = 43 minutes/month)
- **Burn rate** — how fast the error budget is being consumed. A burn rate of 1 means the budget is consumed at exactly the rate that would exhaust it at month-end. A burn rate of 60 means it will be exhausted in 12 hours.

#### Code Example

```javascript
const promClient = require('prom-client');
promClient.collectDefaultMetrics({ prefix: 'node_' }); // CPU, memory, GC, event loop

// ── RED Metrics ────────────────────────────────────────────────────────────
const httpRequestsTotal = new promClient.Counter({
  name: 'http_requests_total',
  help: 'Total number of HTTP requests',
  labelNames: ['method', 'route', 'status_code'],
});

const httpRequestDuration = new promClient.Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2, 5, 10],
});

// RED middleware — instrument every route automatically
app.use((req, res, next) => {
  const end = httpRequestDuration.startTimer();

  res.on('finish', () => {
    const route = req.route?.path ?? req.path ?? 'unknown';
    const labels = { method: req.method, route, status_code: res.statusCode };
    httpRequestsTotal.inc(labels);
    end(labels);
  });

  next();
});

// ── USE Metrics — DB connection pool ──────────────────────────────────────
const dbPoolActive = new promClient.Gauge({
  name: 'db_pool_active_connections',
  help: 'Number of active DB connections',
});
const dbPoolWaiting = new promClient.Gauge({
  name: 'db_pool_waiting_count',
  help: 'Number of requests waiting for a DB connection',
});
const dbPoolUtilization = new promClient.Gauge({
  name: 'db_pool_utilization',
  help: 'DB pool utilization (active / max)',
});

setInterval(() => {
  // pg-pool exposes these
  dbPoolActive.set(pool.totalCount - pool.idleCount);
  dbPoolWaiting.set(pool.waitingCount);
  dbPoolUtilization.set((pool.totalCount - pool.idleCount) / pool.options.max);
  // Alert when utilization > 0.8 (80% of pool in use)
}, 5000);

// ── USE Metrics — Event Loop ───────────────────────────────────────────────
const { performance } = require('perf_hooks');
const eventLoopUtilGauge = new promClient.Gauge({
  name: 'node_event_loop_utilization_ratio',
  help: 'Event loop utilization (0-1)',
});

let prevELU = performance.eventLoopUtilization();
setInterval(() => {
  const curr = performance.eventLoopUtilization(prevELU);
  prevELU = performance.eventLoopUtilization();
  eventLoopUtilGauge.set(curr.utilization);
  // > 0.85 = event loop is saturated
}, 5000);

// ── Prometheus /metrics endpoint ──────────────────────────────────────────
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', promClient.register.contentType);
  res.end(await promClient.register.metrics());
});
```

#### SLO Calculation & Error Budget Burn Rate

```
SLI:  Percentage of HTTP requests with status < 500 AND latency_seconds < 0.5
SLO:  99.9% over a 30-day rolling window
Error budget: 0.1% = 43.8 minutes of allowed failure per month

PromQL to compute SLI compliance:
  sum(rate(http_requests_total{status_code!~"5.."}[30d]))
  /
  sum(rate(http_requests_total[30d]))

Burn rate alert (fires when budget will exhaust in < 1 hour):
  ALERT SloBudgetExhausting
  IF (1 - sli_compliance) / 0.001 > 720   # burn rate > 720x = budget gone in 1h
  FOR 2m
  LABELS { severity = "critical" }
  ANNOTATIONS { summary = "Error budget burning at {{ $value }}x normal rate" }
```

#### Do's and Don'ts

**Do's**
- Alert on burn rate, not raw error rate — a spike of 10 errors/min is alarming for a 100 req/min service but irrelevant for a 1M req/min service. Burn rate normalises for traffic.
- Set SLOs based on p99 latency, not average — averages mask tail latencies that affect 1% of users (potentially 10,000 users at HighLevel's scale).
- Review SLOs quarterly — an SLO never breached might be set too loose. Tighten it to reflect actual user expectations.

**Don'ts**
- Never set an SLO of 100% — it eliminates the error budget and prevents any risk-taking (deployments, experiments, migrations). 99.9% gives 43 minutes of budget per month.
- Don't alert on every SLI breach — alert only when the burn rate indicates meaningful budget exhaustion. Noisy alerts lead to alert fatigue.
- Avoid using average latency as your SLI — outliers are invisible in the mean. Use histogram quantiles (p95, p99).

### **47. Custom Objects & Dynamic Schema Design**

> **Level:** Advanced

#### Why This Is Asked
This is the core domain of the role. "Custom Objects" in a CRM means tenants define their own entity types with custom fields — similar to Salesforce Custom Objects. Designing this correctly is the central system design challenge and the most likely deep-dive topic in the HighLevel interview.

#### Explanation

The fundamental problem: 1 million tenants each want to define their own entity types ("Deal", "Property Listing", "Work Order") with different sets of fields. The platform must store these records, allow filtering on any user-defined field, and report on them — all without knowing the schema in advance.

**Design options:**

1. **EAV (Entity-Attribute-Value)** — one row per field value. Querying multiple fields requires self-joins that become exponentially slow. ❌ Not viable at scale.

2. **JSONB (PostgreSQL)** — one row per record, fields stored as a JSON column with a GIN index. Good up to ~10M records per tenant. Degrades under heavy write load when many unique keys exist.

3. **MongoDB document** — natural fit: schema-less documents, compound indexes on nested `data.*` fields. Most likely HighLevel's primary store. Requires careful index management.

4. **Hybrid: Firestore + ClickHouse** — Firestore for hot transactional data (single record lookups, real-time), ClickHouse for analytics (aggregations, reports, exports). Data flows from Firestore → Pub/Sub → ClickHouse.

The **Schema Registry** is a critical component: it stores field definitions per tenant per object type. The object store validates writes against the registry and the registry determines which fields get indexed.

#### Code Example

```javascript
// ── Schema Registry (MongoDB) ──────────────────────────────────────────────
// Collection: object_schemas
// {
//   tenantId: "abc123",
//   objectType: "deal",
//   version: 3,
//   fields: [
//     { key: "title",      type: "text",   required: true,  filterable: true  },
//     { key: "value",      type: "number", required: false, filterable: true  },
//     { key: "stage",      type: "enum",   required: true,  filterable: true,
//       options: ["New","Qualified","Proposal","Closed Won","Closed Lost"] },
//     { key: "closeDate",  type: "date",   required: false, filterable: true  },
//     { key: "notes",      type: "text",   required: false, filterable: false },
//   ],
//   createdAt, updatedAt
// }

const ObjectSchema = require('./models/ObjectSchema');
const CustomObject = require('./models/CustomObject');

// ── Dynamic field validation on write ─────────────────────────────────────
async function createCustomObject(tenantId, objectType, data) {
  const schema = await ObjectSchema.findOne({ tenantId, objectType });
  if (!schema) throw new AppError(404, `Object type '${objectType}' not found`);

  // Validate required fields
  const errors = [];
  for (const field of schema.fields) {
    if (field.required && data[field.key] == null) {
      errors.push(`Field '${field.key}' is required`);
    }
    if (data[field.key] != null) {
      const typeError = validateFieldType(field, data[field.key]);
      if (typeError) errors.push(typeError);
    }
  }
  if (errors.length) throw new AppError(400, 'Validation failed', errors);

  // Strip unknown fields (fields not in schema)
  const cleanData = {};
  for (const field of schema.fields) {
    if (data[field.key] !== undefined) cleanData[field.key] = coerce(field, data[field.key]);
  }

  return CustomObject.create({ tenantId, objectType, data: cleanData });
}

function validateFieldType(field, value) {
  if (field.type === 'number' && typeof value !== 'number') return `'${field.key}' must be a number`;
  if (field.type === 'date' && isNaN(Date.parse(value))) return `'${field.key}' must be a valid date`;
  if (field.type === 'enum' && !field.options.includes(value)) {
    return `'${field.key}' must be one of: ${field.options.join(', ')}`;
  }
  return null;
}

// ── MongoDB index management (run when schema is created/updated) ──────────
async function syncIndexesForSchema(schema) {
  const collection = mongoose.connection.collection('custom_objects');
  const filterableFields = schema.fields.filter((f) => f.filterable);

  for (const field of filterableFields) {
    // Compound index: tenantId + objectType + the specific filterable field
    const indexKey = {
      tenantId: 1,
      objectType: 1,
      [`data.${field.key}`]: field.type === 'text' ? 'text' : 1,
    };
    await collection.createIndex(indexKey, {
      name: `idx_${schema.tenantId}_${schema.objectType}_${field.key}`,
      background: true, // non-blocking in MongoDB 4.2+
    });
  }
}

// ── Dynamic query builder ──────────────────────────────────────────────────
async function queryCustomObjects(tenantId, objectType, filters, sort, pagination) {
  const schema = await ObjectSchema.findOne({ tenantId, objectType });
  if (!schema) throw new AppError(404, 'Object type not found');

  const schemaFieldMap = Object.fromEntries(schema.fields.map((f) => [f.key, f]));
  const query = { tenantId, objectType };

  // Build filter — only allow filtering on filterable fields
  for (const [key, value] of Object.entries(filters)) {
    const fieldDef = schemaFieldMap[key];
    if (!fieldDef) throw new AppError(400, `Unknown field: ${key}`);
    if (!fieldDef.filterable) throw new AppError(400, `Field '${key}' is not filterable`);
    query[`data.${key}`] = typeof value === 'object' ? value : { $eq: value };
  }

  const total = await CustomObject.countDocuments(query);
  const records = await CustomObject
    .find(query)
    .sort(sort ? { [`data.${sort.field}`]: sort.direction === 'asc' ? 1 : -1 } : { createdAt: -1 })
    .skip((pagination.page - 1) * pagination.pageSize)
    .limit(pagination.pageSize)
    .lean();

  return { total, records };
}

// ── Recommended Architecture ───────────────────────────────────────────────
//
//  Schema Registry (MongoDB)
//    └── field definitions per tenant+objectType
//        └── which fields are filterable → drives index creation
//
//  Object Store (MongoDB)
//    └── { tenantId, objectType, data: {...userDefinedFields} }
//        └── indexes: tenantId+objectType+data.<filterableField>
//
//  Analytics (ClickHouse)
//    └── denormalised wide table: all known fields flattened
//        └── fed via Firestore→Pub/Sub→ClickHouse streaming pipeline
//
//  Search (optional — Elasticsearch)
//    └── for full-text search across all fields of any type
```

#### Key Design Decisions for the Interview Discussion

| Decision | Option A | Option B | HighLevel Recommendation |
|---|---|---|---|
| Schema validation | Write-time (strict) | Read-time (flexible) | Write-time — prevents garbage data, easier reporting |
| Multi-tenancy isolation | Row-level (`tenantId` on every doc) | Collection-per-tenant | Row-level — simpler operations, scalable to 1M tenants |
| Index strategy | Index all fields | Index only `filterable: true` fields | Filterable-only — limits index sprawl, tenant controls it |
| Field deletion | Hard delete from schema | Soft delete (hidden) | Soft delete — preserves existing records, allows undo |
| Analytics | MongoDB aggregations | ClickHouse | ClickHouse for any GROUP BY / COUNT / SUM at scale |

#### Do's and Don'ts

**Do's**
- Enforce a maximum number of filterable fields per tenant (e.g., 20) — MongoDB has a 64-index limit per collection and unbounded indexes degrade write performance.
- Cache schema definitions in Redis (TTL 60 s) — schema reads happen on every write; hitting MongoDB for every record creation at scale is expensive.
- Version schemas — when a tenant changes a field definition, record the old version so existing data can still be read correctly.

**Don'ts**
- Don't use EAV at HighLevel's scale — multi-field queries require exponential self-joins.
- Never store computed or derived values in the custom object record — compute them at read time or in ClickHouse. Storing them creates consistency issues when source data changes.
- Don't allow unlimited text fields on non-searchable objects to be indexed — text indexes in MongoDB are expensive and affect write throughput.

- A tenant has 5 million records of type "Deal". They want to filter by `region` (a new field they just added). How do you add the index without taking the system down?
- How would you design the ClickHouse schema for custom objects when each tenant has different fields?

### **48. MongoDB Performance & Indexing — Deep Dive**

> **Level:** Advanced

#### Why This Is Asked
HighLevel stores hundreds of millions of contacts, leads, and CRM records in MongoDB. An unindexed query on a large collection can take seconds, which multiplied across millions of requests degrades the entire platform. Lead engineers are expected to design indexes proactively and diagnose slow queries systematically.

#### Explanation

**The ESR Rule** — the optimal compound index field order is: **E**quality fields first, then **S**ort fields, then **R**ange fields. This ensures the index can satisfy the equality predicates fully, then the sort (avoiding an in-memory sort), then the range scan — minimising the number of index entries scanned.

**Covered Queries** — a query is "covered" when every field it needs (both in the filter and the projection) is in the index. MongoDB can satisfy it entirely from the index without fetching documents from disk — extremely fast for read-heavy paths.

**Partial Indexes** — index only a subset of documents matching a filter expression. A partial index on `{status: 'active'}` is far smaller than a full index, has lower write overhead, and is faster for queries that always include the filtered condition.

**The Aggregation Pipeline** has specific performance rules: `$match` and `$sort` must come before `$lookup` and `$unwind` to leverage indexes. `$project` should come last to reduce document size only after all filtering is done.

#### Code Example

```javascript
// ── ESR Rule: compound index field ordering ────────────────────────────────
// Query: { tenantId: "abc", status: "active", createdAt: { $gte: date } }
// Sort:  { createdAt: -1 }

// WRONG order: Range before Sort
db.contacts.createIndex({ tenantId: 1, createdAt: -1, status: 1 });
// MongoDB must scan many entries across tenants+dates, then filter by status

// CORRECT ESR order: Equality (tenantId, status) → Sort (createdAt) → Range (implicit via sort)
db.contacts.createIndex({ tenantId: 1, status: 1, createdAt: -1 });
// MongoDB uses equality fields to narrow to one tenant+status bucket,
// then the index is already sorted by createdAt — no in-memory sort needed

// ── Covered query ──────────────────────────────────────────────────────────
// Only fetches email and name — both in the index
db.contacts.createIndex({ tenantId: 1, email: 1, name: 1 });

const contacts = await db.contacts
  .find({ tenantId: 'abc' }, { email: 1, name: 1, _id: 0 })
  .hint({ tenantId: 1, email: 1, name: 1 }); // force the index
// explain() will show: "totalDocsExamined": 0 — pure index scan, no doc fetch

// ── Partial index — only active contacts ──────────────────────────────────
db.contacts.createIndex(
  { tenantId: 1, email: 1 },
  {
    name: 'idx_active_contacts_email',
    partialFilterExpression: { status: 'active' },
    // This index is 5–10× smaller than a full index on large datasets with many churned contacts
  }
);
// Only used by queries that include { status: 'active' } in the filter

// ── explain() — always verify index usage ─────────────────────────────────
const explanation = await db.contacts
  .find({ tenantId: 'abc', status: 'active' })
  .sort({ createdAt: -1 })
  .limit(20)
  .explain('executionStats');

// What to look for:
console.log({
  stage: explanation.executionStats.executionStages.stage,
  // IXSCAN = index used (good), COLLSCAN = full collection scan (bad)
  totalDocsExamined: explanation.executionStats.totalDocsExamined,
  totalKeysExamined: explanation.executionStats.totalKeysExamined,
  // Ratio: keysExamined / docsExamined should be close to 1:1 for efficient queries
  executionTimeMillis: explanation.executionStats.executionTimeMillis,
});

// ── Aggregation pipeline: $match FIRST ────────────────────────────────────

// WRONG: $lookup before $match — joins ALL documents then filters
db.orders.aggregate([
  { $lookup: { from: 'products', localField: 'productId', foreignField: '_id', as: 'product' } },
  { $match: { tenantId: 'abc', status: 'paid' } }, // too late — full collection joined
]);

// CORRECT: $match first, uses index, minimal documents flow through pipeline
db.orders.aggregate([
  { $match: { tenantId: 'abc', status: 'paid' } },       // 1. filter with index
  { $sort: { createdAt: -1 } },                           // 2. sort (uses same index)
  { $limit: 100 },                                        // 3. limit EARLY
  { $lookup: {                                            // 4. join only 100 docs
    from: 'products',
    localField: 'productId',
    foreignField: '_id',      // must have an index on _id in products collection
    as: 'product',
  }},
  { $project: { _id: 1, total: 1, 'product.name': 1 } }, // 5. project last
]);

// ── Slow query log: find queries taking > 100 ms ───────────────────────────
// In mongod.conf:
// operationProfiling:
//   slowOpThresholdMs: 100
//   mode: slowOp

// Programmatically:
await db.command({ profile: 1, slowms: 100 });

// Query the profiler:
const slowOps = await db.system.profile
  .find({ millis: { $gt: 100 } })
  .sort({ ts: -1 })
  .limit(20)
  .toArray();

// Common patterns to look for in slow ops:
// - "nscanned" / "nreturned" ratio > 100:1 → missing index
// - "planSummary": "COLLSCAN" → no index at all
// - "keysExamined" >> "docsExamined" → index exists but selectivity is poor
```

#### Common Performance Pitfalls

| Pitfall | Detection | Fix |
|---|---|---|
| Missing compound index | `explain()` shows COLLSCAN | Add index following ESR rule |
| Index not used due to negation `$ne`, `$nin` | Large `docsExamined` | Restructure query to use positive conditions |
| `$where` with JS function | `explain()` shows `$where` | Replace with native MongoDB operators |
| Large `$in` array (> 1,000 values) | Slow query log | Batch into multiple smaller queries |
| `$lookup` on unindexed `foreignField` | Slow `$lookup` stage | Index the `foreignField` in the joined collection |
| `countDocuments()` on 100M+ collection | Slow counts | Use `estimatedDocumentCount()` or maintain a counter doc |
| No read preference for analytics | Primary overloaded | Use `{ readPreference: 'secondaryPreferred' }` for reporting |

#### Do's and Don'ts

**Do's**
- Run `explain('executionStats')` on every new query before deploying — make sure it shows `IXSCAN`, not `COLLSCAN`.
- Build indexes in the background (`{ background: true }`) in MongoDB 4.2 or use rolling index builds in 4.4+ to avoid blocking reads.
- Use the MongoDB Atlas Performance Advisor or the `$indexStats` aggregation stage to find indexes that are never used and drop them — unused indexes slow down writes.

**Don'ts**
- Don't index low-cardinality fields (like `status` with 3 values) alone — the index won't reduce the scan enough. Combine with a high-cardinality field like `tenantId`.
- Avoid `$regex` queries without anchoring to the start (`/^prefix/`) — an unanchored regex cannot use a B-tree index and degrades to a collection scan.
- Never run index creation in production without testing the performance impact on write throughput first.

- How do you add an index on a 100-million-document collection in a live system without causing downtime?

### **49. ClickHouse for Analytics Workloads**

> **Level:** Intermediate

#### Why This Is Asked
HighLevel processes 1.5 billion messages and 200 million leads per month. Querying aggregations on this volume in MongoDB or PostgreSQL takes seconds to minutes. ClickHouse is purpose-built for this — it returns aggregated results on billions of rows in milliseconds. A Lead Engineer working on the CRM stack must know when and how to use it.

#### Explanation

ClickHouse is a **columnar OLAP** (Online Analytical Processing) database. Unlike row-based databases (MongoDB, PostgreSQL) that store all columns of a row together, ClickHouse stores each column separately. For a query like `SELECT COUNT(*), SUM(value) FROM deals WHERE tenant_id = X`, it only reads the `tenant_id`, and `value` columns — skipping the rest entirely. This makes analytics queries 10–100× faster on large datasets.

Key ClickHouse design principles:
- **Bulk inserts only** — ClickHouse optimises for large batch writes (100+ rows per insert). Single-row inserts bypass its merge optimisation and are very slow.
- **Immutable data** — mutations (`UPDATE`, `DELETE`) are expensive async operations. Design data models to be append-only.
- **Denormalise** — ClickHouse joins are slow and should be avoided. Flatten related data into wide tables before inserting.
- **MergeTree engine family** — the primary storage engine. `ReplacingMergeTree` enables upsert semantics; `SummingMergeTree` pre-aggregates on merge.

#### Code Example

```javascript
const { createClient } = require('@clickhouse/client');

const ch = createClient({
  host: process.env.CLICKHOUSE_HOST,
  username: process.env.CLICKHOUSE_USER,
  password: process.env.CLICKHOUSE_PASSWORD,
  database: 'analytics',
  clickhouse_settings: {
    async_insert: 1,          // buffer inserts server-side
    wait_for_async_insert: 0, // fire-and-forget (higher throughput)
  },
});

// ── Table schema (run once) ────────────────────────────────────────────────
// CREATE TABLE contact_events (
//   tenant_id     String,
//   contact_id    String,
//   event_type    LowCardinality(String), -- enum-like field, stored efficiently
//   channel       LowCardinality(String),
//   occurred_at   DateTime,
//   value         Nullable(Float64),
//   metadata      String                  -- JSON blob for flexible extra fields
// ) ENGINE = MergeTree()
//   PARTITION BY toYYYYMM(occurred_at)    -- partition by month for efficient pruning
//   ORDER BY (tenant_id, occurred_at)     -- primary sort key
//   TTL occurred_at + INTERVAL 2 YEAR;   -- auto-delete data older than 2 years

// ── Streaming ingestion pipeline (Firestore → Pub/Sub → ClickHouse) ────────
const batchBuffer = [];
let flushTimer = null;

async function queueForClickHouse(event) {
  batchBuffer.push({
    tenant_id: event.tenantId,
    contact_id: event.contactId,
    event_type: event.type,
    channel: event.channel ?? 'unknown',
    occurred_at: new Date(event.timestamp),
    value: event.value ?? null,
    metadata: JSON.stringify(event.metadata ?? {}),
  });

  if (batchBuffer.length >= 1000) {
    await flushToClickHouse();
  } else if (!flushTimer) {
    // Flush after 1 second even if batch isn't full
    flushTimer = setTimeout(flushToClickHouse, 1000);
  }
}

async function flushToClickHouse() {
  clearTimeout(flushTimer);
  flushTimer = null;
  if (!batchBuffer.length) return;

  const batch = batchBuffer.splice(0);
  try {
    await ch.insert({
      table: 'contact_events',
      values: batch,
      format: 'JSONEachRow',
    });
    logger.info({ count: batch.length }, 'Flushed batch to ClickHouse');
  } catch (err) {
    logger.error({ err, count: batch.length }, 'ClickHouse insert failed — requeueing');
    batchBuffer.unshift(...batch); // requeue for retry
  }
}

// ── Analytics queries ──────────────────────────────────────────────────────

// Query 1: Event breakdown by channel over last 30 days
async function getEventBreakdown(tenantId) {
  const result = await ch.query({
    query: `
      SELECT
        toStartOfDay(occurred_at) AS day,
        channel,
        event_type,
        count()              AS event_count,
        sum(value)           AS total_value
      FROM contact_events
      WHERE tenant_id = {tenantId: String}
        AND occurred_at >= now() - INTERVAL 30 DAY
      GROUP BY day, channel, event_type
      ORDER BY day DESC, event_count DESC
    `,
    query_params: { tenantId },
    format: 'JSONEachRow',
  });
  return result.json();
}

// Query 2: Funnel analysis — how many contacts progressed through stages
async function getStageFunnel(tenantId, fromDate, toDate) {
  const result = await ch.query({
    query: `
      SELECT
        event_type,
        uniqExact(contact_id) AS unique_contacts
      FROM contact_events
      WHERE tenant_id = {tenantId: String}
        AND occurred_at BETWEEN {from: DateTime} AND {to: DateTime}
        AND event_type IN ('lead_created', 'qualified', 'proposal_sent', 'deal_closed')
      GROUP BY event_type
      ORDER BY unique_contacts DESC
    `,
    query_params: { tenantId, from: fromDate, to: toDate },
    format: 'JSONEachRow',
  });
  return result.json();
}
```

#### ClickHouse vs MongoDB vs PostgreSQL

| Feature | ClickHouse | MongoDB | PostgreSQL |
|---|---|---|---|
| Storage model | Columnar | Document (row-like) | Row-based |
| Best for | Analytics, aggregations, reporting | CRUD, flexible schema, real-time | Transactions, complex joins |
| `GROUP BY` on 1B rows | < 1 second | Minutes (or OOM) | Minutes |
| Single record lookup | Slow (columnar is bad for point reads) | Fast | Fast |
| Write pattern | Bulk inserts (100+ rows) | Single or bulk | Single or bulk |
| Mutations | Expensive, async | Normal | Normal |
| JOIN performance | Slow — avoid/denormalise | `$lookup` OK for small sets | Good for normalized schemas |

#### Do's and Don'ts

**Do's**
- Always batch inserts — minimum 100 rows, ideally 1,000–10,000. Use a buffer+timer approach (see `queueForClickHouse` above).
- Use `LowCardinality(String)` for columns with fewer than ~10,000 distinct values (channel, event_type, status) — it uses dictionary encoding and is significantly faster.
- Partition by date (`PARTITION BY toYYYYMM(...)`) and include the date in WHERE clauses — ClickHouse skips entire partitions it doesn't need.

**Don'ts**
- Don't use ClickHouse for real-time single-record lookups — that's what MongoDB/Firestore is for.
- Avoid `ALTER TABLE ... UPDATE` or `ALTER TABLE ... DELETE` in ClickHouse unless absolutely necessary — mutations are async and expensive. Design for append-only.
- Don't run large ClickHouse queries on the same node that handles production OLTP traffic — dedicate separate replicas for analytics workloads.

### **50. Kubernetes Health Probes, Resources & Autoscaling**

> **Level:** Intermediate

#### Why This Is Asked
HighLevel runs on GKE (Google Kubernetes Engine). A Node.js service with misconfigured health probes causes unnecessary pod restarts during slow DB connections. Under-provisioned resource limits cause OOM kills under load. A Lead Engineer is responsible for the Kubernetes configuration of their service — not just the application code.

#### Explanation

Kubernetes uses three types of health probes:
- **Liveness probe** — "Is the process alive?" If it fails, Kubernetes *restarts* the pod. Use only for genuine process-level failures (infinite loop, deadlock, permanent DB disconnect). Do NOT check external dependencies here — a Redis outage should not restart all your pods.
- **Readiness probe** — "Is the pod ready to serve traffic?" If it fails, the pod is *removed from the load balancer* but not restarted. Use this to check if dependencies are available. During startup or after a DB reconnect, returning 503 here protects users from hitting a pod that can't serve.
- **Startup probe** — "Is the app still initialising?" Disables the liveness probe until startup completes. Critical for apps with slow startup (loading large configs, warming caches) that would otherwise be killed by liveness before they're ready.

**Resource requests vs limits:**
- `requests` — the amount Kubernetes reserves for scheduling. Set this to your typical steady-state usage.
- `limits` — the maximum. For CPU: Node.js is throttled (event loop slows) when it exceeds the CPU limit. For memory: the pod is OOM-killed when it exceeds the memory limit. Set limits generously but not without bound.

#### Code Example

```javascript
// ── Express health endpoints ───────────────────────────────────────────────
let isShuttingDown = false;

// Liveness — only fail if the process itself is broken
// Do NOT check DB, Redis, or external services here
app.get('/healthz/live', (req, res) => {
  if (isShuttingDown) {
    return res.status(503).json({ status: 'shutting down' });
  }
  res.json({ status: 'ok', uptime: process.uptime() });
});

// Readiness — fail when the service cannot correctly serve requests
let dbReady = false;
let redisReady = false;

app.get('/healthz/ready', async (req, res) => {
  if (isShuttingDown) {
    return res.status(503).json({ status: 'shutting down' });
  }

  const checks = await Promise.allSettled([
    db.command({ ping: 1 }).then(() => { dbReady = true; }),
    redis.ping().then(() => { redisReady = true; }),
  ]);

  const failed = checks.filter((c) => c.status === 'rejected');
  if (failed.length) {
    return res.status(503).json({
      status: 'not ready',
      checks: { db: dbReady, redis: redisReady },
    });
  }

  res.json({ status: 'ready', checks: { db: true, redis: true } });
});

// Graceful shutdown — flip readiness to 503 first so LB drains traffic
process.on('SIGTERM', async () => {
  logger.info('SIGTERM received — starting graceful shutdown');
  isShuttingDown = true;

  // Give the load balancer 5 s to stop sending traffic (readiness returns 503)
  await new Promise((r) => setTimeout(r, 5000));

  server.close(async () => {
    await db.close();
    await redis.quit();
    process.exit(0);
  });

  // Hard timeout: force exit if graceful shutdown takes > 25 s
  setTimeout(() => process.exit(1), 25_000).unref();
});
```

# ── Kubernetes Deployment YAML ─────────────────────────────────────────────
apiVersion: apps/v1
kind: Deployment
metadata:
  name: crm-service
spec:
  replicas: 3
  template:
    spec:
      terminationGracePeriodSeconds: 30  # must be > your shutdown timeout (25s above)
      containers:
      - name: crm-service
        image: gcr.io/highlevel/crm-service:latest

        # Resource sizing for a mid-traffic Node.js service
        resources:
          requests:
            cpu: "250m"       # typical steady-state CPU
            memory: "256Mi"   # typical steady-state memory
          limits:
            cpu: "1000m"      # 1 vCPU limit — above this Node.js is throttled
            memory: "512Mi"   # OOM kill if exceeded — must be > Node.js max heap

        # Startup probe — gives app 60 s (12 failures × 5 s) to start
        startupProbe:
          httpGet: { path: /healthz/live, port: 3000 }
          initialDelaySeconds: 5
          periodSeconds: 5
          failureThreshold: 12

        # Liveness — restart if process is broken
        livenessProbe:
          httpGet: { path: /healthz/live, port: 3000 }
          periodSeconds: 10
          failureThreshold: 3

        # Readiness — remove from LB if dependencies unavailable
        readinessProbe:
          httpGet: { path: /healthz/ready, port: 3000 }
          periodSeconds: 5
          failureThreshold: 2

      - type: Pods

    scaleUp:

      policies:
      - type: Pods
        value: 4                       # add at most 4 pods per minute
        periodSeconds: 60

#### Node.js Memory Sizing Rule

```
Node.js heap size is controlled by --max-old-space-size (default: ~1.5 GB on 64-bit)
Kubernetes memory limit must be > --max-old-space-size + off-heap memory

For a service with --max-old-space-size=384:
  memory limit = 384 MB (heap) + 100 MB (off-heap, buffers, OS) = ~512 MB minimum

In Dockerfile:
CMD ["node", "--max-old-space-size=384", "dist/server.js"]
```

#### Do's and Don'ts

**Do's**
- Separate liveness and readiness checks — they have different failure responses (restart vs drain traffic). Combining them causes unnecessary pod restarts during dependency outages.
- Set `terminationGracePeriodSeconds` greater than your shutdown timeout — if it's less, Kubernetes sends SIGKILL before your code finishes cleanup.
- Use the `scaleDown.stabilizationWindowSeconds` on HPA — without it, pods scale down and back up during normal traffic fluctuations (flapping).

**Don'ts**
- Don't check external dependencies (DB, Redis, third-party APIs) in the liveness probe — a Redis outage would trigger a rolling restart of all your pods, making the outage worse.
- Avoid setting `cpu.limits` much higher than `requests` — when a pod bursts to the limit, its CPU is throttled, which causes event loop lag and latency spikes.
- Never run Node.js containers as root — use `runAsNonRoot: true` and a dedicated `node` user in the Dockerfile.

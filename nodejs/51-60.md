### **51. Batching, Concurrency Control & Backpressure**

> **Level:** Advanced

#### Why This Is Asked

Lead roles at companies like HighLevel deal with thousands of concurrent CRM record mutations. Interviewers need to know you can prevent the event loop from becoming a bottleneck — whether through batching DB writes, capping in-flight promises, or respecting stream backpressure.

#### Explanation

**N+1 Problem & DataLoader**

When resolving a list of entities that each need a DB lookup you accumulate O(n) round-trips. DataLoader deduplicates and batches those calls within a single event-loop tick using its scheduling queue.

**p-limit / Semaphore**

`Promise.all` on an unbounded array fires every promise simultaneously, exhausting connection pools and CPU. A semaphore caps in-flight promises to a safe maximum.

**Backpressure in Writable Streams**

`writable.write()` returns `false` when the internal buffer is full. Ignoring this causes unbounded memory growth. The correct pattern is to pause the source and resume on `drain`.

**BatchWriter Pattern**

Collect individual write calls into a buffer; flush on interval *or* max-size, whichever fires first. This turns O(n) individual DB operations into O(1) bulk operations.

#### Code Example

```javascript
// ── 1. DataLoader — batch + deduplicate DB calls ──────────────────────────
const DataLoader = require('dataloader');
const { db } = require('./db');

const contactLoader = new DataLoader(
  async (ids) => {
    // Single query for up to 1 000 ids per tick
    const rows = await db.collection('contacts')
      .find({ _id: { $in: ids } })
      .toArray();

    const map = new Map(rows.map(r => [r._id.toString(), r]));
    return ids.map(id => map.get(id.toString()) ?? new Error(`Not found: ${id}`));
  },
  { maxBatchSize: 1000, cache: true }
);

// ── 2. p-limit — cap concurrent promises ──────────────────────────────────
const pLimit = require('p-limit');

async function sendBulkNotifications(userIds) {
  const limit = pLimit(20); // max 20 in-flight at once
  const tasks = userIds.map(id =>
    limit(() => notificationService.send(id))
  );
  const results = await Promise.allSettled(tasks);
  const failed = results.filter(r => r.status === 'rejected');
  if (failed.length) console.error(`${failed.length} notifications failed`);
}

// ── 3. Semaphore class ─────────────────────────────────────────────────────
class Semaphore {
  constructor(max) {
    this._max = max;
    this._count = 0;
    this._queue = [];
  }

  acquire() {
    if (this._count < this._max) {
      this._count++;
      return Promise.resolve();
    }
    return new Promise(resolve => this._queue.push(resolve));
  }

  release() {
    this._count--;
    if (this._queue.length) {
      this._count++;
      this._queue.shift()();
    }
  }
}

const sem = new Semaphore(5);
async function throttled(fn) {
  await sem.acquire();
  try { return await fn(); }
  finally { sem.release(); }
}

// ── 4. BatchWriter — flush on interval OR size ────────────────────────────
class BatchWriter {
  constructor({ flushInterval = 500, maxSize = 500, flush }) {
    this._buffer = [];
    this._flush = flush;
    this._timer = setInterval(() => this._doFlush(), flushInterval);
    this._timer.unref(); // don't keep process alive
    this._maxSize = maxSize;
  }

  write(item) {
    this._buffer.push(item);
    if (this._buffer.length >= this._maxSize) this._doFlush();
  }

  async _doFlush() {
    if (!this._buffer.length) return;
    const batch = this._buffer.splice(0);
    try { await this._flush(batch); }
    catch (err) { console.error('BatchWriter flush failed', err); }
  }

  async close() {
    clearInterval(this._timer);
    await this._doFlush();
  }
}

const writer = new BatchWriter({
  flushInterval: 200,
  maxSize: 1000,
  flush: async (docs) => db.collection('events').insertMany(docs),
});

// ── 5. Stream backpressure ─────────────────────────────────────────────────
const { Readable, Writable } = require('stream');

function pipeWithBackpressure(readable, writable) {
  readable.on('data', (chunk) => {
    const canContinue = writable.write(chunk);
    if (!canContinue) {
      readable.pause();
      writable.once('drain', () => readable.resume());
    }
  });
  readable.on('end', () => writable.end());
}
```

#### Do's and Don'ts

**Do's**
- Use DataLoader when a request can trigger repeated lookups for the same entity type
- Set `maxBatchSize` to avoid single queries with 100 k IDs
- Use `Promise.allSettled` (not `Promise.all`) when partial failure is acceptable
- Unref interval timers in long-lived workers so they don't block process exit
- Flush BatchWriter on SIGTERM before the process exits

**Don'ts**
- Don't use `Promise.all` on unbounded arrays against a connection-pooled DB
- Don't ignore the return value of `writable.write()` in high-throughput pipelines
- Don't share a single DataLoader instance across requests (stale cache); create per-request loaders
- Don't rely solely on intervals for flushing; always flush on max-size too to bound latency

#### Common Follow-up Questions

1. How does DataLoader's per-tick batching interact with async/await — why does batching only happen within one microtask checkpoint?
2. How would you implement a priority queue in front of a Semaphore so that critical operations bypass lower-priority work?
3. What happens to items in a BatchWriter's buffer if the Node process crashes mid-interval? How do you make it durable?

### **52. Event-Driven Architecture with GCP Pub/Sub**

> **Level:** Advanced

#### Why This Is Asked

HighLevel's Custom Objects service communicates across microservices asynchronously. Pub/Sub is the backbone. Interviewers probe whether you understand durability guarantees, ordering, at-least-once vs exactly-once, and operational concerns like dead-letter topics.

#### Explanation

**Core Model**

A *publisher* pushes messages to a *topic*. One or more *subscriptions* fan out copies. A *subscriber* receives messages either via *push* (HTTP webhook Pub/Sub calls) or *pull* (subscriber polls). Messages are acknowledged (`ack`) or negatively acknowledged (`nack`). Unacked messages are redelivered up to the subscription's `ackDeadline`.

**Message Ordering**

By default Pub/Sub does not guarantee order. Setting an *ordering key* on messages forces all messages with the same key to be delivered in order on a given subscriber, but you must enable `enableMessageOrdering` on the subscription client.

**Exactly-Once Delivery**

Pub/Sub offers at-least-once by default. Exactly-once delivery (available on some subscription types) uses acknowledgement IDs that are tied to a delivery attempt; duplicate deliveries are suppressed by the service for a short window.

**Dead-Letter Topics**

After `maxDeliveryAttempts` failed acks, Pub/Sub forwards the message to a configured dead-letter topic. A separate subscriber processes the DLT for alerting, manual replay, or dropping.

**Push vs Pull**

| | Push | Pull |
|---|---|---|
| Latency | Lower (server-initiated) | Slightly higher (client polls) |
| Scaling | Pub/Sub scales push — beware thundering herd | Client controls rate |
| Auth | Pub/Sub must reach your endpoint | Simpler — just needs IAM |
| Best for | Webhooks, low-volume triggers | High-throughput, batch workers |

#### Code Example

```javascript
const { PubSub } = require('@google-cloud/pubsub');

const pubsub = new PubSub({ projectId: process.env.GCP_PROJECT });

// ── Publisher ──────────────────────────────────────────────────────────────
async function publishContactUpdated(contact) {
  const topic = pubsub.topic('contact-events', {
    // Batch: flush every 10 ms or when 100 messages queued
    batching: { maxMessages: 100, maxMilliseconds: 10 },
  });

  const messageId = await topic.publishMessage({
    json: { type: 'CONTACT_UPDATED', payload: contact },
    orderingKey: contact.tenantId, // order within a tenant
    attributes: { schemaVersion: '1', source: 'contacts-service' },
  });

  return messageId;
}

// ── Subscriber (pull) ──────────────────────────────────────────────────────
function startSubscriber() {
  const subscription = pubsub.subscription('contact-events-worker', {
    flowControl: { maxMessages: 50 }, // cap in-flight messages
    enableMessageOrdering: true,
  });

  subscription.on('message', async (message) => {
    try {
      const event = JSON.parse(message.data.toString());
      await handleContactEvent(event);
      message.ack();
    } catch (err) {
      console.error('Handler failed', err);
      // nack causes redelivery; only nack for transient errors
      if (isTransient(err)) {
        message.nack();
      } else {
        // Permanent failure — ack to prevent infinite loop; DLT handles it
        message.ack();
      }
    }
  });

  subscription.on('error', (err) => {
    console.error('Subscription error', err);
  });
}

// ── Dead-letter processor ──────────────────────────────────────────────────
function startDltProcessor() {
  const dltSub = pubsub.subscription('contact-events-dlt');
  dltSub.on('message', async (message) => {
    const deliveryAttempt = parseInt(
      message.attributes['googclient_deliveryattempt'] ?? '0'
    );
    console.error('DLT message', {
      messageId: message.id,
      deliveryAttempt,
      data: message.data.toString(),
    });
    await alertingService.notify({ severity: 'high', message });
    message.ack();
  });
}

// ── Context propagation across Pub/Sub boundary ────────────────────────────
const { propagation, context } = require('@opentelemetry/api');

async function publishWithTrace(topic, payload) {
  const carrier = {};
  propagation.inject(context.active(), carrier);
  await topic.publishMessage({
    json: payload,
    attributes: carrier, // W3C traceparent rides as a message attribute
  });
}

function extractTraceFromMessage(message) {
  const carrier = message.attributes ?? {};
  return propagation.extract(context.active(), carrier);
}
```

#### Do's and Don'ts

**Do's**
- Set `flowControl.maxMessages` to prevent one slow subscriber from buffering unlimited messages in memory
- Use ordering keys scoped to a tenant/entity, not globally — global ordering keys serialize all messages through one publisher
- Configure a dead-letter topic from day one; retrofitting it later is painful
- Publish with `batching` enabled to improve throughput; tune `maxMilliseconds` for acceptable latency
- Propagate OTel trace context through message attributes

**Don'ts**
- Don't nack a message indefinitely for a permanent error (bad schema, deleted entity) — it will loop forever until the DLT catches it
- Don't rely on message order without `enableMessageOrdering` on the subscription client
- Don't put large blobs in message data; keep messages under 10 MB and reference GCS for payloads
- Don't use the same subscription for multiple incompatible consumers — give each consumer its own subscription on the topic

#### Common Follow-up Questions

1. How do you replay messages from a dead-letter topic into the main topic after fixing a bug?
2. Pub/Sub guarantees at-least-once delivery. How do you make your handlers idempotent?
3. How would you handle backpressure when a downstream DB is slow and Pub/Sub is delivering faster than you can process?

### **53. Saga Pattern & Distributed Transactions**

> **Level:** Advanced

#### Why This Is Asked

Custom Objects in HighLevel involve creating records that span multiple services (CRM, notifications, audit log, search index). Without a distributed transaction protocol, partial failures leave data inconsistent. The Saga pattern is the industry-standard answer for microservices.

#### Explanation

A **Saga** is a sequence of local transactions, each of which publishes an event or calls the next step. If any step fails, previously completed steps are undone via **compensating transactions**.

**Choreography Saga**

Each service listens for events and decides what to do. No central coordinator. Services publish success/failure events that trigger the next service.

- Pros: loose coupling, no single point of failure
- Cons: hard to track overall flow state, debugging is distributed across logs

**Orchestration Saga**

A central **Saga Orchestrator** service tells each participant what to do via direct calls or commands. It tracks state in its own DB.

- Pros: flow visible in one place, easier to add steps
- Cons: introduces a central coordinator that can become a bottleneck

**Compensating Transactions**

Each forward step must have a defined compensating (rollback) step that undoes its effect. Compensations must themselves be idempotent because they may run more than once.

| Forward Step | Compensating Step |
|---|---|
| `createContact` | `deleteContact` |
| `chargePayment` | `refundPayment` |
| `sendWelcomeEmail` | *(not reversible — mark as voided in audit)* |

#### Code Example

```javascript
// ── Orchestration Saga — Custom Object Creation ───────────────────────────

class CreateCustomObjectSaga {
  constructor({ db, pubsub, searchService }) {
    this.db = db;
    this.pubsub = pubsub;
    this.searchService = searchService;
  }

  async execute(payload) {
    const sagaId = crypto.randomUUID();
    const log = [];

    try {
      // Step 1: persist to MongoDB
      const record = await this.db.collection('custom_objects').insertOne({
        ...payload,
        _sagaId: sagaId,
        createdAt: new Date(),
      });
      log.push({ step: 'db_insert', id: record.insertedId });

      // Step 2: index in search service
      await this.searchService.index({
        id: record.insertedId.toString(),
        ...payload,
      });
      log.push({ step: 'search_index', id: record.insertedId });

      // Step 3: publish domain event
      await this.pubsub.topic('custom-object-events').publishMessage({
        json: { type: 'CUSTOM_OBJECT_CREATED', sagaId, payload },
      });
      log.push({ step: 'event_published' });

      return { ok: true, id: record.insertedId };
    } catch (err) {
      // Run compensations in reverse order
      await this._compensate(log, err);
      throw err;
    }
  }

  async _compensate(log, reason) {
    console.error('Saga compensating due to', reason.message);
    for (const entry of log.reverse()) {
      try {
        if (entry.step === 'db_insert') {
          await this.db.collection('custom_objects').deleteOne({ _id: entry.id });
        }
        if (entry.step === 'search_index') {
          await this.searchService.delete(entry.id.toString());
        }
        // event_published: cannot unsend; publish a CANCELLED event instead
        if (entry.step === 'event_published') {
          await this.pubsub.topic('custom-object-events').publishMessage({
            json: { type: 'CUSTOM_OBJECT_CREATION_FAILED', reason: reason.message },
          });
        }
      } catch (compErr) {
        // Compensation itself failed — log for manual intervention
        console.error('COMPENSATION FAILED', { step: entry.step, compErr });
      }
    }
  }
}

// ── Choreography Saga — via events ────────────────────────────────────────

// contacts-service publishes:
// { type: 'CONTACT_CREATED', contactId, tenantId }

// billing-service listens and publishes on success:
// { type: 'CONTACT_BILLING_ACTIVATED', contactId }
// or on failure:
// { type: 'CONTACT_BILLING_FAILED', contactId, reason }

// contacts-service listens to BILLING_FAILED and compensates:
// deleteContact(contactId)
```

#### Do's and Don'ts

**Do's**
- Persist saga state to a DB before starting each step so you can resume after a crash
- Make every compensating transaction idempotent (use a `sagaId` as idempotency key)
- Use orchestration for complex multi-step sagas (>3 steps) to keep the flow visible
- Use choreography for simple two-service interactions where coupling is a concern
- Publish a `SAGA_FAILED` event to a monitoring topic so ops teams can detect stuck sagas

**Don'ts**
- Don't use database transactions spanning multiple services — that's 2PC and it doesn't scale
- Don't assume compensations will always succeed — design for manual intervention fallback
- Don't emit compensating events before completing the compensating action — the action must complete first
- Don't skip persisting intermediate saga state; a process restart mid-saga must be recoverable

#### Common Follow-up Questions

1. How do you handle a saga that is stuck mid-way because a downstream service is down for hours?
2. What is the difference between a compensating transaction and a rollback in ACID terms?
3. How would you test a saga's compensation path in integration tests?

### **54. Idempotency in Distributed APIs**

> **Level:** Advanced

#### Why This Is Asked

In distributed systems, network failures cause retries, and retries cause duplicate requests. A Lead Engineer must design APIs that produce the same result regardless of how many times the same request is delivered — a foundational property for payment, CRM mutation, and event-processing endpoints.

#### Explanation

**Idempotency Key**

The client generates a unique key (UUID) per logical operation and sends it as a header (`Idempotency-Key`). The server stores the result of the first execution and returns the cached response for any duplicate request with the same key.

**Storage Strategy**

Results are stored in Redis with a TTL (typically 24 hours). The key space is `idempotency:{clientId}:{idempotencyKey}`. Before executing, the server checks if the key exists:
- **Not found** → execute, store result, return
- **Found with result** → return cached result immediately
- **Found with `processing` marker** → another instance is executing; return 409 or wait

**At-Least-Once Delivery**

Message queues guarantee *at-least-once* delivery. Consumer handlers must be idempotent using the message ID or a domain-specific identifier as a deduplication key.

**Non-idempotent side-effects**

Some actions (emails, push notifications) cannot be truly undone. The workaround is to record a "sent" marker in a DB before sending and check it on re-entry.

#### Code Example

```javascript
// ── Express middleware for idempotency keys ────────────────────────────────
const redis = require('ioredis');
const client = new redis(process.env.REDIS_URL);

const IDEMPOTENCY_TTL = 60 * 60 * 24; // 24 hours

async function idempotencyMiddleware(req, res, next) {
  const key = req.headers['idempotency-key'];
  if (!key) return next(); // optional for GET-like mutations

  const redisKey = `idempotency:${req.user.tenantId}:${key}`;
  const existing = await client.get(redisKey);

  if (existing === 'processing') {
    return res.status(409).json({
      error: 'Request is currently being processed. Please retry in a moment.',
    });
  }

  if (existing) {
    const cached = JSON.parse(existing);
    return res.status(cached.status).json(cached.body);
  }

  // Mark as in-progress with short TTL
  await client.set(redisKey, 'processing', 'EX', 30);

  // Capture response to store it
  const originalJson = res.json.bind(res);
  res.json = async (body) => {
    const result = JSON.stringify({ status: res.statusCode, body });
    // Only cache successful or client-error responses; not 5xx
    if (res.statusCode < 500) {
      await client.set(redisKey, result, 'EX', IDEMPOTENCY_TTL);
    } else {
      await client.del(redisKey); // allow retry on server error
    }
    return originalJson(body);
  };

  next();
}

// ── Idempotent Pub/Sub message handler ────────────────────────────────────
async function handleContactCreated(message) {
  const { contactId, tenantId } = JSON.parse(message.data.toString());
  const dedupKey = `msg:processed:${message.id}`;

  const alreadyProcessed = await client.set(
    dedupKey, '1', 'EX', 86400, 'NX' // SET if Not eXists
  );

  if (!alreadyProcessed) {
    // Duplicate delivery — skip processing
    message.ack();
    return;
  }

  try {
    await createSearchIndex(contactId, tenantId);
    message.ack();
  } catch (err) {
    await client.del(dedupKey); // allow reprocessing on failure
    message.nack();
  }
}

// ── Idempotent DB upsert ───────────────────────────────────────────────────
async function upsertContact(contact) {
  await db.collection('contacts').updateOne(
    { externalId: contact.externalId, tenantId: contact.tenantId },
    {
      $set: { ...contact, updatedAt: new Date() },
      $setOnInsert: { createdAt: new Date() },
    },
    { upsert: true }
  );
}
```

#### Do's and Don'ts

**Do's**
- Scope idempotency keys to a tenant/user (`tenantId:key`) to prevent cross-tenant key collisions
- Use `SET NX` (set-if-not-exists) in Redis atomically to prevent race conditions between concurrent requests
- Delete the Redis key (allow retry) when the handler throws a 5xx error
- Cache the exact response status + body, not just a flag, so the client gets back the original response
- Set a bounded TTL — idempotency keys are not permanent records

**Don'ts**
- Don't use the same idempotency key for different logical operations
- Don't forget to clean up the `processing` marker if your handler crashes (use short TTL for it)
- Don't make idempotency keys case-sensitive unless you clearly document it
- Don't rely solely on DB unique constraints for idempotency — they throw errors instead of returning cached responses

#### Common Follow-up Questions

1. How do you handle idempotency for a multi-step saga where partial completion is possible?
2. What TTL would you choose for an idempotency key in a payment system vs a notification system, and why?
3. How would you implement idempotency without Redis (e.g., using only PostgreSQL or MongoDB)?

### **55. Multi-Tenancy Architecture Patterns**

> **Level:** Advanced

#### Why This Is Asked

HighLevel is a multi-tenant SaaS platform serving tens of thousands of agencies. A Lead Engineer must understand the data isolation, performance, and operational tradeoffs of different multi-tenancy models at the DB, service, and API layer.

#### Explanation

**Three Models**

| Model | Isolation | Cost | Complexity |
|---|---|---|---|
| **Silo** (one DB per tenant) | Strongest | Highest | Simple per-tenant ops |
| **Bridge** (one schema per tenant, shared DB) | Medium | Medium | Schema migrations complex |
| **Pool** (shared tables, `tenantId` column) | Weakest | Lowest | Requires row-level filtering everywhere |

Most large SaaS platforms use a **pool model with silo escape hatches** — shared infra for the majority of tenants, with dedicated resources for enterprise accounts.

**Row-Level Security**

In pool model, every query must include `tenantId`. Missing a filter leaks data across tenants. The safest pattern is a **TenantDB wrapper** that injects `tenantId` into all queries automatically.

**Connection Pooling per Tenant**

In pool model with a shared connection pool, a noisy tenant can exhaust the pool. Mitigations: per-tenant connection pool caps, query timeouts, and rate limiting at the API layer.

**Schema Versioning per Tenant**

Enterprise tenants often need custom fields, workflows, or data retention policies. The schema registry pattern (see Q47) enables per-tenant schema customisation without proliferating DB schemas.

#### Code Example

```javascript
// ── TenantDB — injects tenantId into all queries ───────────────────────────
class TenantDB {
  constructor(db, tenantId) {
    this._db = db;
    this._tenantId = tenantId;
  }

  collection(name) {
    const col = this._db.collection(name);
    return new TenantCollection(col, this._tenantId);
  }
}

class TenantCollection {
  constructor(col, tenantId) {
    this._col = col;
    this._tenantId = tenantId;
  }

  _withTenant(filter = {}) {
    return { ...filter, tenantId: this._tenantId };
  }

  find(filter, options) {
    return this._col.find(this._withTenant(filter), options);
  }

  findOne(filter, options) {
    return this._col.findOne(this._withTenant(filter), options);
  }

  insertOne(doc) {
    return this._col.insertOne({ ...doc, tenantId: this._tenantId });
  }

  updateOne(filter, update, options) {
    return this._col.updateOne(this._withTenant(filter), update, options);
  }

  deleteOne(filter) {
    return this._col.deleteOne(this._withTenant(filter));
  }

  countDocuments(filter) {
    return this._col.countDocuments(this._withTenant(filter));
  }
}

// Usage in a request handler — tenantDb is scoped to the authenticated tenant
async function getContactHandler(req, res) {
  const tenantDb = new TenantDB(mongoClient.db(), req.tenant.id);
  const contact = await tenantDb.collection('contacts').findOne({ _id: req.params.id });
  if (!contact) return res.status(404).json({ error: 'Not found' });
  res.json(contact);
}

// ── Per-tenant rate limiting (token bucket via Redis) ──────────────────────
async function tenantRateLimiter(req, res, next) {
  const key = `ratelimit:${req.tenant.id}`;
  const limit = req.tenant.plan === 'enterprise' ? 10000 : 1000; // req/min
  const now = Date.now();
  const window = 60_000;

  const [count] = await redis.multi()
    .incr(key)
    .pexpire(key, window)
    .exec();

  if (count[1] > limit) {
    return res.status(429).json({
      error: 'Rate limit exceeded',
      retryAfter: Math.ceil(window / 1000),
    });
  }
  next();
}

// ── Compound index for pool model — always lead with tenantId ──────────────
// db.contacts.createIndex({ tenantId: 1, email: 1 }, { unique: true })
// db.contacts.createIndex({ tenantId: 1, createdAt: -1 })
// db.custom_objects.createIndex({ tenantId: 1, objectType: 1, _id: 1 })
```

#### Do's and Don'ts

**Do's**
- Always lead compound indexes with `tenantId` in pool-model collections
- Use a TenantDB/TenantCollection wrapper so tenant isolation is enforced at the data layer, not scattered across handlers
- Log a warning (or throw) when a query is constructed without a `tenantId` filter
- Implement per-tenant rate limiting and per-tenant connection pool caps to prevent noisy neighbour issues
- Test for tenant data leakage explicitly in integration tests

**Don'ts**
- Don't rely on application-level code to always remember to add `tenantId` — enforce it structurally
- Don't store tenant-specific secrets (API keys, OAuth tokens) as plain fields in shared documents — use a dedicated secrets store per tenant
- Don't apply the same query timeout to enterprise and free-tier tenants — enterprise plans often need longer timeouts for large exports
- Don't share caches across tenants without scoping cache keys by tenantId

### **56. CQRS Pattern in Node.js**

> **Level:** Advanced

#### Why This Is Asked

Custom Objects at HighLevel need to handle high write throughput (mutations) and complex read queries (analytics, filtered lists) simultaneously. CQRS decouples the write model from the read model, allowing each to scale and evolve independently.

#### Explanation

**Command Query Responsibility Segregation (CQRS)**

The write side (Commands) mutates state and publishes domain events. The read side (Queries) maintains one or more denormalised *read models* optimised for specific query patterns.

**Event Sourcing (optional companion)**

Instead of storing only current state, the system stores the full history of events. Current state is derived by replaying events. This gives a complete audit log and the ability to rebuild any read model from scratch.

**Read Model Synchronisation**

A projection worker subscribes to domain events and updates the read model. This is eventually consistent — there is a lag between a write and the updated read model becoming visible.

**When to use CQRS**

- High read-to-write ratios where read queries are complex
- When read and write models have different scalability needs
- When you need multiple representations of the same data (search index, analytics, API)

#### Code Example

```javascript
// ── Command side — CustomObjectService ────────────────────────────────────
class CustomObjectCommandService {
  constructor({ db, eventBus }) {
    this.db = db;
    this.eventBus = eventBus;
  }

  async createObject(tenantId, objectType, fields) {
    const id = new ObjectId();
    const now = new Date();

    // Write to the write model (append-only event store)
    await this.db.collection('object_events').insertOne({
      _id: new ObjectId(),
      aggregateId: id,
      type: 'OBJECT_CREATED',
      tenantId,
      objectType,
      fields,
      version: 1,
      timestamp: now,
    });

    // Publish to the event bus for projection workers
    await this.eventBus.publish('custom-object-events', {
      type: 'OBJECT_CREATED',
      aggregateId: id.toString(),
      tenantId,
      objectType,
      fields,
      timestamp: now.toISOString(),
    });

    return id;
  }

  async updateField(tenantId, objectId, fieldKey, fieldValue) {
    const latestVersion = await this._getVersion(objectId);

    await this.db.collection('object_events').insertOne({
      aggregateId: new ObjectId(objectId),
      type: 'FIELD_UPDATED',
      tenantId,
      fieldKey,
      fieldValue,
      version: latestVersion + 1,
      timestamp: new Date(),
    });

    await this.eventBus.publish('custom-object-events', {
      type: 'FIELD_UPDATED',
      aggregateId: objectId,
      tenantId,
      fieldKey,
      fieldValue,
    });
  }

  async _getVersion(objectId) {
    const latest = await this.db.collection('object_events')
      .find({ aggregateId: new ObjectId(objectId) })
      .sort({ version: -1 })
      .limit(1)
      .next();
    return latest?.version ?? 0;
  }
}

// ── Projection worker — builds the read model ──────────────────────────────
class CustomObjectProjection {
  constructor({ db, subscription }) {
    this.db = db;
    this.subscription = subscription;
  }

  start() {
    this.subscription.on('message', async (message) => {
      const event = JSON.parse(message.data.toString());
      try {
        await this._project(event);
        message.ack();
      } catch (err) {
        console.error('Projection failed', err);
        message.nack();
      }
    });
  }

  async _project(event) {
    const col = this.db.collection('custom_objects_read');
    if (event.type === 'OBJECT_CREATED') {
      await col.insertOne({
        _id: new ObjectId(event.aggregateId),
        tenantId: event.tenantId,
        objectType: event.objectType,
        fields: event.fields,
        createdAt: new Date(event.timestamp),
        updatedAt: new Date(event.timestamp),
      });
    }
    if (event.type === 'FIELD_UPDATED') {
      await col.updateOne(
        { _id: new ObjectId(event.aggregateId) },
        {
          $set: {
            [`fields.${event.fieldKey}`]: event.fieldValue,
            updatedAt: new Date(),
          },
        }
      );
    }
  }
}

// ── Query side — fast reads from the denormalised read model ───────────────
class CustomObjectQueryService {
  constructor({ db }) {
    this.db = db;
  }

  async listByType(tenantId, objectType, { page = 1, limit = 50 } = {}) {
    return this.db.collection('custom_objects_read')
      .find({ tenantId, objectType })
      .sort({ createdAt: -1 })
      .skip((page - 1) * limit)
      .limit(limit)
      .toArray();
  }
}
```

#### Do's and Don'ts

**Do's**
- Keep commands and queries in separate service classes so they can be scaled independently
- Make projection handlers idempotent (use `upsert` rather than `insert`) — events may be replayed
- Version your events so that schema changes to events don't break older projections
- Use read models tailored to specific query patterns — don't try to make one read model serve all queries
- Expose a `/rebuild-projection` admin endpoint for replaying all events when you add a new projection

**Don'ts**
- Don't query the write model (event store) from the read API — that defeats the purpose of CQRS
- Don't implement CQRS for simple CRUD services; it adds significant complexity for minimal gain
- Don't let the projection lag become unbounded — monitor projection lag as an operational metric
- Don't store derived/computed fields only in the read model without a way to recompute them

#### Common Follow-up Questions

1. How do you handle a schema change in events that breaks an existing projection?
2. How do you rebuild a projection from scratch in a live production system without downtime?
3. What is the maximum acceptable projection lag for a CRM UI, and how would you alert on it?

### **57. API Gateway & Backend-for-Frontend (BFF) Pattern**

> **Level:** Advanced

#### Why This Is Asked

HighLevel serves multiple clients — web dashboard, mobile apps, third-party integrations — each with different data needs. An API Gateway centralises cross-cutting concerns and a BFF pattern gives each client an optimised API surface without burdening downstream microservices.

#### Explanation

**API Gateway Responsibilities**

- Authentication & authorisation (JWT validation, OAuth token exchange)
- Rate limiting per client / tenant
- Request routing to downstream microservices
- Response aggregation (fan-out to multiple services, merge results)
- Protocol translation (REST-to-gRPC, WebSocket-to-SSE)
- Observability (centrally attach tracing headers)

**Backend for Frontend (BFF)**

One API Gateway serving all clients forces a lowest-common-denominator API. The BFF pattern creates a thin, client-specific aggregation layer for each client type (web, mobile, partner). Each BFF owns its own API contract and aggregates calls to shared microservices.

```
Mobile App → Mobile BFF → [contacts-service, deals-service, notifications-service]
Web App    → Web BFF    → [contacts-service, schema-registry, analytics-service]
Partners   → Public API → [contacts-service, webhooks-service] (rate-limited)
```

**When to use BFF**

Use BFF when different clients need fundamentally different shapes of the same data. Avoid it if you only have one client — it adds a deployment unit and a latency hop.

#### Code Example

```javascript
// ── Web BFF — aggregates contact detail page ───────────────────────────────
const express = require('express');
const axios = require('axios');
const { propagation, context } = require('@opentelemetry/api');

const app = express();

// Fetch contact + related deals + custom objects in parallel
app.get('/bff/contacts/:id/detail', async (req, res) => {
  const { id } = req.params;
  const tenantId = req.headers['x-tenant-id'];
  const authHeader = req.headers['authorization'];

  // Propagate trace context downstream
  const carrier = {};
  propagation.inject(context.active(), carrier);
  const headers = { authorization: authHeader, 'x-tenant-id': tenantId, ...carrier };

  try {
    const [contact, deals, customObjects] = await Promise.allSettled([
      axios.get(`${process.env.CONTACTS_SVC}/contacts/${id}`, { headers }),
      axios.get(`${process.env.DEALS_SVC}/deals?contactId=${id}`, { headers }),
      axios.get(`${process.env.OBJECTS_SVC}/objects?linkedContactId=${id}`, { headers }),
    ]);

    res.json({
      contact: contact.status === 'fulfilled' ? contact.value.data : null,
      deals: deals.status === 'fulfilled' ? deals.value.data : [],
      customObjects: customObjects.status === 'fulfilled' ? customObjects.value.data : [],
      _partial: [contact, deals, customObjects].some(r => r.status === 'rejected'),
    });
  } catch (err) {
    res.status(502).json({ error: 'Upstream service unavailable' });
  }
});

// ── API Gateway middleware — JWT validation + rate limiting ────────────────
const jwt = require('jsonwebtoken');
const rateLimit = require('express-rate-limit');

function gatewayAuth(req, res, next) {
  const token = req.headers['authorization']?.replace('Bearer ', '');
  if (!token) return res.status(401).json({ error: 'Missing token' });
  try {
    req.claims = jwt.verify(token, process.env.JWT_PUBLIC_KEY, {
      algorithms: ['RS256'],
    });
    req.tenant = { id: req.claims.tenantId, plan: req.claims.plan };
    next();
  } catch {
    res.status(401).json({ error: 'Invalid token' });
  }
}

const gatewayRateLimiter = rateLimit({
  windowMs: 60_000,
  max: (req) => req.tenant?.plan === 'enterprise' ? 10000 : 1000,
  keyGenerator: (req) => req.tenant?.id ?? req.ip,
  handler: (req, res) => res.status(429).json({ error: 'Rate limit exceeded' }),
});

// ── Response caching at the gateway ───────────────────────────────────────
async function cacheMiddleware(req, res, next) {
  if (req.method !== 'GET') return next();
  const key = `cache:${req.tenant.id}:${req.path}:${JSON.stringify(req.query)}`;
  const cached = await redis.get(key);
  if (cached) {
    res.setHeader('X-Cache', 'HIT');
    return res.json(JSON.parse(cached));
  }
  const originalJson = res.json.bind(res);
  res.json = async (body) => {
    if (res.statusCode === 200) await redis.set(key, JSON.stringify(body), 'EX', 30);
    return originalJson(body);
  };
  next();
}
```

#### Do's and Don'ts

**Do's**
- Use `Promise.allSettled` in BFF aggregation so one failing upstream doesn't 502 the entire page
- Propagate distributed trace context from gateway to all downstream calls
- Keep BFFs thin — aggregation and response shaping only; no business logic
- Cache GET responses at the gateway for short TTLs (5–30 s) to reduce downstream load
- Validate JWT at the gateway so individual microservices don't need auth logic

**Don'ts**
- Don't put business logic in the BFF — it becomes a monolith disguised as a gateway
- Don't create a BFF per feature — group by client type (web, mobile, partner)
- Don't forward raw downstream errors to clients — normalise error shapes at the gateway
- Don't forget to propagate the tenant context to downstream calls in the aggregation layer

#### Common Follow-up Questions

1. How do you version a BFF API when the mobile app can't always be updated immediately?
2. What are the deployment and operational implications of having three BFFs vs one API gateway?
3. How would you handle WebSocket connections through a BFF running as multiple pods in Kubernetes?

### **58. Node.js Cluster Mode & PM2 in Production**

> **Level:** Advanced

#### Why This Is Asked

Node.js is single-threaded. On a multi-core machine, one Node process leaves most CPU cores idle. Cluster mode and PM2 solve this, but they introduce nuances around state sharing, IPC, graceful reloads, and health monitoring that a Lead Engineer must understand deeply.

#### Explanation

**Cluster Module**

The built-in `cluster` module lets a master process `fork()` worker processes that all share the same server port via OS-level socket sharing. The master distributes incoming connections to workers using round-robin (default on Linux).

**PM2**

PM2 is a process manager that wraps cluster mode with production features: automatic restarts on crash, log aggregation, `ecosystem.config.js` for deployment config, graceful reload (zero downtime), and a monitoring dashboard.

**Graceful Reload**

`pm2 reload` sends `SIGINT` to workers one by one, waiting for them to drain in-flight requests before restarting. Workers must listen for `SIGINT`/`SIGTERM` and stop accepting new connections, then close after in-flight requests complete.

**Shared State Problem**

Workers are separate processes — they do not share memory. In-memory caches, rate limit counters, and session stores must live in Redis, not in process memory.

**Worker Communication (IPC)**

Workers can communicate with the master (and each other via the master) using `process.send()` and `process.on('message')`. Use this sparingly — IPC is synchronous-ish and adds latency.

#### Code Example

```javascript
// ── ecosystem.config.js — PM2 production config ────────────────────────────
module.exports = {
  apps: [{
    name: 'api-server',
    script: './src/server.js',
    instances: 'max',          // one worker per CPU core
    exec_mode: 'cluster',
    max_memory_restart: '512M',
    listen_timeout: 5000,       // ms to wait for worker to report ready
    kill_timeout: 10000,        // ms to wait for graceful shutdown before SIGKILL
    env: { NODE_ENV: 'development' },
    env_production: { NODE_ENV: 'production', PORT: 3000 },
    log_date_format: 'YYYY-MM-DD HH:mm:ss',
    error_file: './logs/err.log',
    out_file: './logs/out.log',
    merge_logs: true,
  }],
};

// ── server.js — graceful shutdown ──────────────────────────────────────────
const http = require('http');
const app = require('./app');

const server = http.createServer(app);
server.listen(process.env.PORT, () => {
  // Signal PM2 that this worker is ready to accept connections
  if (process.send) process.send('ready');
  console.log(`Worker ${process.pid} listening on ${process.env.PORT}`);
});

let isShuttingDown = false;

async function gracefulShutdown(signal) {
  if (isShuttingDown) return;
  isShuttingDown = true;
  console.log(`Worker ${process.pid} received ${signal}, shutting down`);

  // Stop accepting new connections
  server.close(async () => {
    try {
      // Flush any pending DB writes, close connections
      await mongoClient.close();
      await redisClient.quit();
      console.log(`Worker ${process.pid} shutdown complete`);
      process.exit(0);
    } catch (err) {
      console.error('Shutdown error', err);
      process.exit(1);
    }
  });

  // Force exit if drain takes too long (PM2 kill_timeout handles this)
  setTimeout(() => process.exit(1), 9000);
}

process.on('SIGINT', () => gracefulShutdown('SIGINT'));
process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));

// ── Manual cluster setup (without PM2) ────────────────────────────────────
const cluster = require('cluster');
const os = require('os');

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  console.log(`Primary ${process.pid} forking ${numCPUs} workers`);
  for (let i = 0; i < numCPUs; i++) cluster.fork();
  cluster.on('exit', (worker, code, signal) => {
    console.error(`Worker ${worker.process.pid} died (${signal || code}), restarting`);
    cluster.fork();
  });
} else {
  require('./server');
}
```

#### Do's and Don'ts

**Do's**
- Send `process.send('ready')` after the server is listening so PM2 knows the worker is healthy before routing traffic to it
- Set `kill_timeout` > the longest expected request duration to allow in-flight requests to complete
- Move all shared state (sessions, rate limits, feature flags) to Redis before enabling cluster mode
- Monitor per-worker memory; restart workers that exceed a memory threshold (`max_memory_restart`)
- Use `instances: 'max'` for CPU-bound work and tune down for I/O-bound services if worker startup overhead matters

**Don'ts**
- Don't store session state in process memory in cluster mode — different workers handle different requests
- Don't use `process.exit(0)` immediately on SIGTERM — always drain in-flight connections first
- Don't fork more workers than CPUs; extra workers increase context-switching overhead
- Don't rely on IPC for high-frequency state sync; use Redis instead

#### Common Follow-up Questions

1. How does PM2's `pm2 reload` differ from `pm2 restart`, and when would you use each?
2. In cluster mode, if a worker crashes mid-request, what happens to the client that sent that request?
3. How would you implement per-worker metrics (CPU, memory, event loop lag) and aggregate them at the PM2 master level?

### **59. Chaos Engineering & Resilience Testing**

> **Level:** Advanced

#### Why This Is Asked

At HighLevel scale, failures are not exceptional — they are routine. Interviewers want to know whether you proactively verify that the system behaves correctly under failure, not just under normal load. Chaos engineering is the discipline of deliberately injecting failures to reveal weaknesses before they become incidents.

#### Explanation

**Principles of Chaos Engineering (Netflix)**

1. Build a hypothesis about steady-state behaviour (e.g., "p99 latency stays below 500 ms")
2. Introduce real-world events in a controlled way (network partition, pod kill, CPU stress)
3. Measure whether the steady state holds
4. Fix the weaknesses revealed

**Types of Fault Injection**

| Fault | What it reveals |
|---|---|
| Service crash | Retry logic, circuit breakers |
| Network latency (300 ms–10 s) | Timeout settings, cascading slowness |
| Network partition | Fallback behaviour, data consistency |
| CPU/memory stress | Autoscaling triggers, memory leak detection |
| Dependency outage | Circuit breakers, graceful degradation |
| Message queue backlog | Backpressure, DLT behaviour |

**Tools**

- **Chaos Monkey** (Netflix OSS) — randomly terminates instances in production
- **Chaos Toolkit** — declarative YAML-based fault injection experiments
- **Toxiproxy** (Shopify) — TCP proxy that injects latency, bandwidth limits, connection drops in test environments
- **Gremlin** — commercial chaos-as-a-service platform
- **Litmus / ChaosMesh** — Kubernetes-native chaos operators

#### Code Example

```javascript
// ── Toxiproxy — inject latency in integration tests ────────────────────────
// Start Toxiproxy server: toxiproxy-server
// Create proxy: toxiproxy-cli create mongo-proxy --listen 127.0.0.1:27018 --upstream 127.0.0.1:27017

// Test: system still serves degraded responses under 2 s DB latency
const Toxiproxy = require('toxiproxy-node-client');

describe('Contact service resilience', () => {
  let proxy;
  const toxiproxyClient = new Toxiproxy.Toxiproxy('http://localhost:8474');

  beforeEach(async () => {
    proxy = await toxiproxyClient.get('mongo-proxy');
    await proxy.removeToxic('latency');
  });

  afterEach(async () => {
    await proxy.removeToxic('latency').catch(() => {});
  });

  it('returns cached data when MongoDB has 3 s latency', async () => {
    // Inject 3000 ms latency on the MongoDB proxy
    await proxy.addToxic({
      name: 'latency',
      type: 'latency',
      attributes: { latency: 3000 },
    });

    const response = await request(app).get('/contacts/123').timeout(5000);
    expect(response.status).toBe(200);
    expect(response.headers['x-cache']).toBe('HIT'); // served from cache
  });

  it('opens circuit breaker after 5 consecutive failures', async () => {
    await proxy.addToxic({
      name: 'latency',
      type: 'timeout',
      attributes: { timeout: 0 }, // drop connection immediately
    });

    // Exhaust the circuit breaker threshold
    for (let i = 0; i < 5; i++) {
      await request(app).get('/contacts/123').catch(() => {});
    }

    // Circuit should now be OPEN — fast fail without hitting DB
    const start = Date.now();
    const response = await request(app).get('/contacts/123');
    const elapsed = Date.now() - start;

    expect(response.status).toBe(503);
    expect(elapsed).toBeLessThan(50); // circuit breaker short-circuits
  });
});

// ── Fault injection middleware for lower environments ──────────────────────
function faultInjectionMiddleware(req, res, next) {
  if (process.env.NODE_ENV === 'production') return next();

  const faultHeader = req.headers['x-inject-fault'];
  if (!faultHeader) return next();

  if (faultHeader === 'latency') {
    const ms = parseInt(req.headers['x-fault-latency-ms'] ?? '1000');
    return setTimeout(next, ms);
  }
  if (faultHeader === 'error') {
    return res.status(503).json({ error: 'Injected fault' });
  }
  next();
}
```

#### Do's and Don'ts

**Do's**
- Start chaos experiments in staging before running in production
- Define a clear steady-state hypothesis and success criteria before injecting faults
- Run experiments during business hours when engineers are available to respond
- Have a kill switch — a way to immediately stop a chaos experiment
- Make chaos experiments automated and part of your CI pipeline for lower environments

**Don'ts**
- Don't inject faults in production without alerting and rollback plans
- Don't start with catastrophic faults (full DC outage) — start small (single pod kill, 100 ms latency)
- Don't ignore the blast radius — ensure experiments can't cascade into a multi-service incident
- Don't run chaos experiments during peak traffic windows without explicit approval

#### Common Follow-up Questions

1. How would you convince a risk-averse team to adopt chaos engineering?
2. What metrics would you monitor during a chaos experiment to define "steady state"?
3. How do you automate chaos experiments so they run regularly as part of CI/CD?

### **60. Zero-Downtime Deployments & Database Migrations**

> **Level:** Advanced

#### Why This Is Asked

HighLevel's CRM platform must stay available 24/7. A Lead Engineer needs to ship code changes and database schema changes without taking the system offline or causing errors during the deployment window.

#### Explanation

**Rolling Deployments**

Kubernetes replaces pods one by one. At any moment, old and new code are running simultaneously. This means your new code must be backward-compatible with the old database schema, and vice versa.

**Expand-Contract Pattern (for schema changes)**

A three-phase migration approach that eliminates the need for a maintenance window:

1. **Expand** — Add the new column/index/collection as *optional* (nullable, no FK constraint yet). Deploy code that writes to both old and new structures.
2. **Migrate** — Run a background job to backfill existing data to the new structure.
3. **Contract** — Once all data is migrated and all pods run the new code, remove the old structure.

**Feature Flags**

New code paths are hidden behind a feature flag. Deploy the code but enable the flag for a small percentage of tenants first, then ramp up. Roll back by disabling the flag without redeploying.

**Blue-Green Deployment**

Run two identical environments (Blue = current production, Green = new version). Switch traffic atomically with a load balancer rule. Roll back by switching back to Blue. More expensive (double infra) but instant rollback.

#### Code Example

```javascript
// ── Expand-Contract migration example ─────────────────────────────────────
// SCENARIO: Rename field `phone` → `primaryPhone` in contacts collection.

// Phase 1 — EXPAND: write to BOTH fields in new code
async function createContact(data) {
  return db.collection('contacts').insertOne({
    ...data,
    phone: data.primaryPhone ?? data.phone,         // old field (backward compat)
    primaryPhone: data.primaryPhone ?? data.phone,  // new field
  });
}

// Phase 2 — BACKFILL: migrate existing docs (run as a background job)
async function backfillPrimaryPhone() {
  const cursor = db.collection('contacts')
    .find({ primaryPhone: { $exists: false } })
    .batchSize(500);

  let count = 0;
  for await (const doc of cursor) {
    await db.collection('contacts').updateOne(
      { _id: doc._id },
      { $set: { primaryPhone: doc.phone } }
    );
    count++;
    if (count % 1000 === 0) console.log(`Backfilled ${count} contacts`);
  }
  console.log(`Backfill complete: ${count} contacts updated`);
}

// Phase 3 — CONTRACT: only after all pods run new code and backfill is done
// db.contacts.updateMany({}, { $unset: { phone: '' } }) — run once, carefully

// ── Graceful shutdown for zero-downtime rolling deploy ─────────────────────
// (Kubernetes sends SIGTERM when a pod is being replaced)
// Refer to Q58 for the graceful shutdown implementation.

// ── Readiness probe — tell K8s when the pod is ready to receive traffic ────
app.get('/health/ready', async (req, res) => {
  try {
    await Promise.all([
      mongoClient.db().command({ ping: 1 }),
      redis.ping(),
    ]);
    res.status(200).json({ status: 'ready' });
  } catch {
    res.status(503).json({ status: 'not ready' });
  }
});

// ── Feature flag controlled rollout ───────────────────────────────────────
async function isFeatureEnabled(featureName, tenantId) {
  const flag = await redis.get(`feature:${featureName}`);
  if (!flag) return false;
  const config = JSON.parse(flag);

  if (config.enabled === true) return true;
  if (config.tenants?.includes(tenantId)) return true;

  // Percentage rollout via consistent hashing
  if (config.percentage) {
    const hash = tenantId.split('').reduce((acc, c) => acc + c.charCodeAt(0), 0);
    return (hash % 100) < config.percentage;
  }
  return false;
}

// Usage:
app.post('/contacts', async (req, res) => {
  const useNewSchema = await isFeatureEnabled('new-contact-schema', req.tenant.id);
  const contact = useNewSchema
    ? await newContactService.create(req.body)
    : await legacyContactService.create(req.body);
  res.status(201).json(contact);
});
```

#### Do's and Don'ts

**Do's**
- Always use the expand-contract pattern for schema changes in a live system
- Test rollback: before every deploy, confirm you can roll back the code change *and* the DB migration independently
- Use a readiness probe to ensure pods only receive traffic after they have verified DB and cache connectivity
- Run backfill jobs during off-peak hours and instrument them with progress logging
- Use feature flags to decouple code deploys from feature releases

**Don'ts**
- Don't add a NOT NULL column without a default in a single migration step — it will lock the table in Postgres or fail validation in MongoDB schema mode
- Don't delete old columns in the same deploy that adds new ones — always separate expand and contract by at least one full deploy cycle
- Don't deploy breaking API changes without versioning (`/v2/contacts`) so old clients still work
- Don't assume your migration script is fast — test with production-scale data volumes in staging first
